2. 함수형 인터페이스를 받는 자바 메서드에 직접 객체를 넣는 것과 람다를 사용해서 넣는 것의 차이는 무엇일까요? (p.259) (김세진)

| 구분 | 객체 | 람다 |
| --- | --- | --- |
| 객체 생성 | 항상 생성 | 캡처 없으면 재사용 가능 |
| 코드 길이 | 김 | 짧음 |
| 가독성 | 낮음 | 높음 |
| 성능 | 매번 생성 | 최적화 가능 |

- 자바의 SAM(Single Abstract Method) 인터페이스를 인자로 받는 메서드를 호출할 때 두 방식은 **객체 생성 시점**에서 큰 차이가 납니다.

- **객체 직접 전달 (`new Runnable() { ... }`):** 메서드를 호출할 때마다 **매번 새로운 객체가 생성**됩니다.
- **람다 전달:**
    - **변수를 캡처하지 않을 때:** 프로그램 전체에서 **단 하나의 인스턴스(Singleton)**만 생성하여 재사용합니다. 성능상 이점이 큽니다.
    - **변수를 캡처할 때:** 호출할 때마다 새로운 객체가 생성됩니다

2. with, also, apply 각각의 특징과 언제 사용하게 될까요 (김규)
- with
  - 어떤 객체의 이름을 반복하지 않고도 그 객체에 대해 다양한 연산을 수행
  - 실제로는 파라미터가 두 개인 함수. 첫 번째가 대상이고, 두 번째가 람다일 뿐.
  - 첫 번째 인자로 받은 객체를 두 번째 인자로 받은 람다의 수신 객체로 만듦
  - 불필요한 변수의 반복을 줄일 수 있음
- apply
  - 항상 결과로 수신 객체를 반환함
  - 임의의 타입의 확장 함수로 호출 가능
  - 인스턴스를 만들면서 즉시 프로퍼티 일부를 초기화하는 경우 유용함
- also
  - 수신 객체에 대해 어떤 동작을 수행한 후 수신 객체를 반환
  - 수신 객체를 인자로 참조함 (it 을 써야 함)
  - 객체 자체의 프로퍼티나 함수를 다루는 동작이 아니라, 원래의 수신 객체를 인자로 받는 동작을 실행할 때 유용
  - 의미론적으로 어떤 효과를 추가로 수행하는 것으로 해석할 수 있음

(승수)
1. 코틀린 람다에서 외부 변수 var과 val을 캡처할 때 내부적으로 어떻게 동작하는가? (김규)
- val
  - 변수의 값 자체를 복사하여 기억
- var
  - Ref class instance 안에 var 프로퍼티를 생성해서 값을 넣고, 해당 인스턴스에 대한 참조를 기억

2. 람다가 외부 변수를 캡처하지 않을 때 vs 캡처할 때 익명 객체 생성 차이는? (김세진)
- - **캡처 안 할 때:** 컴파일러가 람다를 `static final` 필드로 정의하여, singleton으로 재사용합니다. (메모리 효율적)
- **캡처 할 때:** 람다가 캡처한 변수를 저장해야 하므로, 호출될 때마다 해당 변수들을 필드로 가진 새로운 익명 클래스 인스턴스를 생성합니다.
  
3. SAM 생성자란 무엇이며 언제 사용해야 하는가? (김세진)
- 함수형 인터페이스를 정의하고 싶을 때
- 하나의 추상 메서드에 다른 비추상 미세드도 정의하고 싶을 때
- 람다를 명시적으로 함수형 인터페이스의 인스턴스로 변환하는 메서드 (예: `Runnable { ... }`)
    - 람다로 생성된 인스턴스를 변수에 저장해두고 재사용해야 할 때.
    - 오버로딩된 메서드 중 특정 인터페이스 타입을 명시적으로 지정해야 할 때.
    - 자바 메서드에 람다를 넘겼는데, 반환 값이 해당 인터페이스 타입이어야 할 때 유용합니다.

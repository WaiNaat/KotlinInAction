# 정리
- 고차 함수
  - 다른 함수를 인자로 받거나, 함수를 반환하는 함수
    - 인자로 받은 함수의 호출은 일반 함수의 호출 구문과 같음
  - 코틀린에서는 람다나 함수 참조를 통해 함수를 값으로 표현
- 함수 타입 
  - 문법: `(Int, String) -> Unit`
  - 파라미터 이름 지정 가능 `(value: Int, name: String) -> Unit`
  - 파라미터 이름은 타입 검사 시 무시됨. 가독성 목적으로 사용.
  - 자바 람다는 자동으로 코틀린 함수 타입으로 변환됨.
- 함수 타입의 구현
  - 코틀린 함수 타입은 `FunctionN` 인터페이스임.
    - N은 파라미터 개수
  - 인터페이스에는 `invoke`라는 유일한 메서드 존재. 이걸 호출하면 함수를 호출함
  - 함수 타입의 변수는 FunctionN 을 구현하는 클래스의 인스턴스
  - FunctionN 은 컴파일러가 생성한 합성 타입. List의 ComponentN 처럼 개수가 정해져 있는 것이 아님.
- 파라미터로 함수 타입 사용하기
  - 기본값 지정 가능
  - nullable type 사용 가능
  - `invoke`도 직접 호출은 가능
- 함수를 반환하는 함수
  - 상태나 다른 조건에 따라 로직이 달라지는 경우 사용할만함
- 람다를 사용하면 데이터의 반복을 줄이고 반복적인 행동을 추출할 수 있음
  - strategy pattern에서 클래스까지 필요하지 않은 경우 람다로 strategy를 표현할 수 있음
- 람다의 성능
  - 코틀린은 보통 람다를 익명 클래스로 컴파일
  - 람다식마다 새로운 클래스를 만들고, 람다가 변수를 캡처한다면 매번 새로운 객체를 생성함
  - `inline` 변경자를 이용하면 이런 부가 비용을 줄일 수 있음
- `inline`
  - 함수를 호출하는 코드를, 함수를 호출하는 바이트코드가 아닌 함수 본문 자체를 바이트코드에 컴파일하는 것
  - 프로퍼티 접근자에도 인라이닝 적용 가능
- 모든 람다를 인라이닝할 수 없음
  - 함수 본문에서 파라미터로 받은 람다를 직접 호출하면 쉽게 가능
  - 인라인 함수의 본문에서 람다식을 바로 호출하거나 다른 인라인 함수에 전달하는 경우 가능
  - 파라미터로 받은 람다를 다른 변수에 저장하고 나중에 그 변수를 사용하는 경우 불가능 (람다를 표현하는 객체를 별도로 저장해야 함)
  - 시퀀스에 사용한 람다는 불가능 (어딘가에 저장했다가 마지막에 몰아서 호출하므로)
- `noinline`
  - 둘 이상의 람다를 인자로 받는 함수에서 일부 람다만 인라인하고 싶을 때 사용
- 언제 인라이닝을 할 것인가
  - 람다를 인자로 받는 함수만 성능이 좋아질 가능성이 높음
    - 일반 함수 호출
      - JVM은 코드 실행을 분석해서 가장 이익이 되는 방향으로 호출을 인라이닝함
      - 코틀린은 바이트코드 복붙이라 코드 중복이 생김
    - 람다를 인자로 받는 함수
      - 함수 호출 비용이 줄어듦
      - 람다를 표현하는 별도의 객체와 인스턴스를 만들 필요 없음
  - non-local return이 필요한 경우
  - 항상 코드 크기에 주의
- non-local return
  - 자신을 둘러싸고 있는 블록보다 더 바깥의 블록을 반환하게 하는 return문
  - 람다를 인자로 받는 함수가 inline fun 일 때만 가능
  - 그게 아닌 람다는 return 사용 자체가 불가능
- local return
  - 람다의 실행을 끝냄
  - label 사용해서 가능
  - 익명 함수로도 가능
    - 람다식에 대한 문법적 편의
    - return의 컨셉 자체가 가장 가까운 fun 을 종료하는 것이므로 local return 가능

# 궁금했던 것들
### 함수 타입의 파라미터는 callback() 처럼 호출 가능하다. nullable 일 경우 callback?.invoke() 처럼 호출이 된다. 왜 자바스크립트처럼 callback?.() 이라는 문법은 만들지 않았을까?
- 컨셉의 차이
- `callback()` 자체가 `callback.invoke()`의 syntatic sugar
- 코틀린의 `?.`은 뒤쪽에 대상자의 이름이 있어야 함

### strategy pattern이란 뭘까?
- strategy interface 를 구현하는 구현체 여러 개
- 상황에 따라 알맞은 strategy를 설정하고 호출하는 context

### non-local return이 유용한 것은 알겠음. 근데 함수가 본인 외부의 세상을 통제한다는 것이 이질적인 컨셉이 아닌지?
- 코틀린 자체의 컨셉이 if, else, while 등 전통적인 제어구조 키워드를 줄이고, 그걸 표준 라이브러리 함수로 대체하려는 것
- 공감은 가지 않는 내용. 이런 컨셉은 라이브러리나 프레임워크가 지향하는 것이지 '언어'가 지향하는 건 어울리지 않다고 생각함. 이럴거면 자바의 대체재를 만들지 말고 본인들만의 새로운 언어를 팠어야 하는 것이 아닌지?

# 📌Question

### Q1) 람다를 인라이닝할 수 있는 조건들에는 어떤 것이 있을까요?
- 가능
  - 함수 본문에서 파라미터로 받은 람다를 직접 호출
  - 인라인 함수의 본문에서 람다식을 바로 호출하거나 다른 인라인 함수에 전달
- 불가능
  - 파라미터로 받은 람다를 다른 변수에 저장하고 나중에 그 변수를 사용 (람다를 표현하는 객체를 별도로 저장해야 함)
  - 시퀀스에 사용한 람다 (어딘가에 저장했다가 마지막에 몰아서 호출하므로)

### Q2) 함수 타입의 `invoke` 메서드는 무엇일까요?
- 코틀린 함수 타입은 `FunctionN` 인터페이스임.
- 인터페이스에는 `invoke`라는 유일한 메서드 존재. 이걸 호출하면 함수를 호출함

### Q1) 코틀린에서 `val f: (Int) -> Int = { it + 1 }` 같은 함수 타입은 컴파일 후 어떻게 변환되는가?
- `Function1<Integer, Integer>` 인터페이스의 구현체로 변환
- 내부적으로 `invoke` 메서드를 통해 함수를 호출
    
### Q2) 인라인 함수가 항상 성능에 이득인가?
- 일반 함수 호출: 큰 이득 없음
  - JVM은 코드 실행을 분석해서 가장 이익이 되는 방향으로 호출을 인라이닝함
  - 코틀린은 바이트코드 복붙이라 코드 중복이 생김
- 람다를 인자로 받는 함수: 성능상 이득
  - 함수 호출 비용이 줄어듦
  - 람다를 표현하는 별도의 객체와 인스턴스를 만들 필요 없음

### Q1) 자바에서 코틀린의 고차 함수를 호출할 때, 반환 타입이 Unit인 경우 주의해야 할 점은 무엇인가요?
- java의 Void와 다르게 kotlin의 Unit은 값이 존재하는 실제 객체이기 때문에 명시적으로 `Unit.INSTANCE`를 반환해야 함
    
### Q2) 컬렉션 연산에서 asSequence를 사용할 때와 일반 연산을 사용할 때, 인라이닝 측면에서 어떤 차이가 있나요?
- sequence 의 연산은 연산 중간과정을 별도로 저장했다가 마지막에 몰아서 호출하기 떄문에, sequence에 사용한 람다는 인라이닝이 불가능

# 10장 고차 함수: 람다를 파라미터와 반환값으로 사용

## 1. 다른 함수를 인자로 받거나 반환하는 함수 정의: 고차 함수

**고차 함수(Higher-Order Function)** 란 다른 함수를 인자로 받거나 함수를 반환하는 함수다.
Kotlin에서는 람다나 함수 참조를 "값"으로 다룰 수 있기 때문에 고차 함수를 자연스럽게 정의할 수 있다.

표준 라이브러리의 `filter`, `map`, `with` 등이 모두 고차 함수다.

```kotlin
list.filter { it > 0 }   // filter는 (T) -> Boolean 함수를 받음
```

### 함수 타입

함수 타입은 **`(파라미터 타입) -> 반환 타입`** 형태로 표현한다.

```kotlin
val sum: (Int, Int) -> Int = { x, y -> x + y }
val action: () -> Unit = { println(42) }
```

일반 함수 정의에서는 `Unit` 반환을 생략할 수 있지만, 함수 타입에서는 생략할 수 없다.

#### Nullable 함수 타입

괄호 위치에 따라 의미가 달라진다.

```kotlin
var canReturnNull: (Int, Int) -> Int? = { _, _ -> null }   // 반환값이 nullable
var funOrNull: ((Int, Int) -> Int)? = null                 // 함수 자체가 nullable
```

#### 파라미터 이름 지정

함수 타입에 파라미터 이름을 줄 수 있다. IDE 자동완성과 가독성에 도움이 된다.

```kotlin
fun performRequest(
    url: String,
    callback: (code: Int, content: String) -> Unit
)

performRequest("...") { code, content -> /* ... */ }
// 호출 시 람다의 파라미터 이름은 자유롭게 바꿔도 됨 (타입 체크에는 영향 없음)
```

### 인자로 전달받은 함수 호출

함수 타입 파라미터는 일반 함수처럼 호출해서 사용한다.

```kotlin
fun twoAndThree(operation: (Int, Int) -> Int) {
    val result = operation(2, 3)
    println("Result is $result")
}

twoAndThree { a, b -> a + b }   // 5
twoAndThree { a, b -> a * b }   // 6
```

#### 실용 예: 직접 만들어보는 `filter`

```kotlin
fun String.filter(predicate: (Char) -> Boolean): String {
    val sb = StringBuilder()
    for (index in indices) {
        val element = get(index)
        if (predicate(element)) sb.append(element)
    }
    return sb.toString()
}

println("ab1c".filter { it in 'a'..'z' })   // "abc"
```

`predicate` 파라미터의 타입은 `(Char) -> Boolean`. 함수처럼 `predicate(element)`로 바로 호출한다.

### 자바에서 코틀린 함수 타입 사용

함수 타입은 컴파일되면 **`FunctionN` 인터페이스**로 변환된다 (`Function0`, `Function1`, ... `Function22`).
각 인터페이스는 `invoke` 메서드 하나를 가진다.

```kotlin
// Kotlin
fun processTheAnswer(f: (Int) -> Int) {
    println(f(42))
}
```

```java
// Java 8+
processTheAnswer(number -> number + 1);   // 람다로 호출 가능
```

Java에서 `Unit` 반환 함수 타입을 호출할 때는 명시적으로 `Unit.INSTANCE`를 반환해야 한다.
Java의 `void`와 Kotlin의 `Unit`은 다르기 때문이다.

```java
// Java
List<String> strings = new ArrayList<>();
strings.add("42");
CollectionsKt.forEach(strings, s -> {
    System.out.println(s);
    return Unit.INSTANCE;   // 필수
});
```

### 디폴트 값과 nullable 함수 타입

라이브러리 API에서 자주 쓰이는 패턴이다.

#### 디폴트 람다 인자

```kotlin
fun <T> Collection<T>.joinToString(
    separator: String = ", ",
    prefix: String = "",
    postfix: String = "",
    transform: (T) -> String = { it.toString() }   // 디폴트 람다
): String {
    val result = StringBuilder(prefix)
    for ((index, element) in this.withIndex()) {
        if (index > 0) result.append(separator)
        result.append(transform(element))
    }
    result.append(postfix)
    return result.toString()
}
```

#### Nullable 함수 타입

```kotlin
fun foo(callback: (() -> Unit)? = null) {
    callback?.invoke()   // 안전 호출 + invoke
}
```

Nullable 함수 타입에서는 `invoke()` 명시 호출이 필수다. `callback?()` 같은 문법은 없다.

### 함수를 함수에서 반환

비즈니스 로직 분기에서 전략을 함수로 캡슐화할 때 유용하다.

```kotlin
enum class Delivery { STANDARD, EXPEDITED }

class Order(val itemCount: Int)

fun getShippingCostCalculator(delivery: Delivery): (Order) -> Double {
    if (delivery == Delivery.EXPEDITED) {
        return { order -> 6 + 2.1 * order.itemCount }
    }
    return { order -> 1.2 * order.itemCount }
}

val calculator = getShippingCostCalculator(Delivery.EXPEDITED)
println("Shipping costs ${calculator(Order(3))}")   // 12.3
```

함수 반환 타입은 그대로 함수 타입(`(Order) -> Double`)을 적어주면 된다.

### 람다로 중복 줄이기

고차 함수의 진짜 가치는 반복되는 코드 패턴을 추상화하는 데 있다.

```kotlin
data class SiteVisit(val path: String, val duration: Double, val os: OS)
enum class OS { WINDOWS, LINUX, MAC, IOS, ANDROID }

// 하드코딩된 필터링 - OS 조건마다 새 함수 필요
fun List<SiteVisit>.averageWindowsDuration() =
    filter { it.os == OS.WINDOWS }.map(SiteVisit::duration).average()

// 일반화된 고차 함수 - 어떤 조건이든 람다로 받음
fun List<SiteVisit>.averageDurationFor(predicate: (SiteVisit) -> Boolean) =
    filter(predicate).map(SiteVisit::duration).average()

log.averageDurationFor { it.os in setOf(OS.ANDROID, OS.IOS) }
log.averageDurationFor { it.os == OS.IOS && it.path == "/signup" }
```

디자인 패턴(특히 Strategy 패턴)이 람다 한 줄로 대체된다.
Java의 익명 클래스 보일러플레이트가 사라진다는 점이 핵심이다.

## 2. 인라인 함수를 사용해 람다의 부가 비용 없애기

람다는 편리하지만 익명 클래스로 컴파일되면서 객체 생성 비용이 발생한다.
변수를 캡처하는 람다는 매 호출마다 새 객체가 만들어질 수도 있다.

`inline` 키워드를 붙이면 컴파일러가 함수 본문을 호출 지점에 그대로 펼쳐넣어 이런 부가 비용을 없앤다.

### 인라이닝이 작동하는 방식

`inline` 함수는 호출되는 모든 지점에서 본문이 그대로 복사된다(바이트코드 수준).

```kotlin
inline fun <T> synchronized(lock: Lock, action: () -> T): T {
    lock.lock()
    try {
        return action()
    } finally {
        lock.unlock()
    }
}

val l = ReentrantLock()
synchronized(l) {
    // 보호된 작업
}
```

위 호출은 컴파일되면 대략 이렇게 펼쳐진다.

```kotlin
l.lock()
try {
    // 보호된 작업          // 람다 본문이 그대로 인라인됨
} finally {
    l.unlock()
}
```

결과적으로 `Function0` 객체 생성도, `invoke` 호출도 없다. 일반 코드처럼 동작한다.

#### 인라인되지 않는 경우

람다를 변수에 담아 넘기면, 그 변수가 가리키는 람다 객체는 호출 지점에 알려져 있지 않으므로 인라인되지 않는다.

```kotlin
class LockOwner(val lock: Lock) {
    fun runUnderLock(body: () -> Unit) {
        synchronized(lock, body)   // body는 인라인되지 않고 객체로 전달됨
    }
}
```

### 인라인 함수의 제약

람다 파라미터를 본문에서 직접 호출하거나, 다른 인라인 함수에 넘기는 경우에만 인라이닝이 가능하다.
람다를 어딘가에 저장하거나 반환하면 인라인할 수 없다.

#### `noinline` — 일부 파라미터만 인라이닝 제외

```kotlin
inline fun foo(inlined: () -> Unit, noinline notInlined: () -> Unit) {
    // inlined는 인라인, notInlined는 일반 객체로 처리
}
```

#### `crossinline` — 비로컬 return 금지

람다가 함수 본문 안에서 다른 실행 컨텍스트(예: 다른 람다, 객체 메서드)로 넘어가야 할 때 사용한다.
호출자에서 비로컬 `return`을 사용할 수 없게 막는다.

```kotlin
inline fun runAsync(crossinline block: () -> Unit) {
    Thread { block() }.start()   // block이 다른 컨텍스트(Runnable)에서 실행됨
}
```

| 키워드 | 의미 |
|---|---|
| `inline` | 함수 본문과 람다를 호출 지점에 펼침 |
| `noinline` | 특정 람다 파라미터만 인라이닝 제외 |
| `crossinline` | 인라인은 하되, 비로컬 return 금지 |

### 컬렉션 연산 인라이닝

표준 라이브러리의 `filter`, `map`, `forEach` 같은 컬렉션 연산은 모두 `inline`으로 정의되어 있다.

```kotlin
public inline fun <T> Iterable<T>.filter(predicate: (T) -> Boolean): List<T> {
    return filterTo(ArrayList<T>(), predicate)
}
```

따라서 다음 코드는 람다 객체나 추가 호출 비용 없이 for 루프와 거의 동일한 성능을 낸다.

```kotlin
people.filter { it.age < 30 }
      .map(Person::name)
```

#### 그러나 중간 컬렉션은 만들어진다

`filter`와 `map`을 연쇄하면 중간 리스트가 만들어진다. 데이터가 크면 `asSequence()`가 답이다.

```kotlin
people.asSequence()
      .filter { it.age < 30 }
      .map(Person::name)
      .toList()
```

다만 시퀀스 연산은 인라인되지 않는다. 시퀀스의 람다는 객체로 저장되어 지연 실행되기 때문이다.
따라서 작은 컬렉션은 그냥 `filter`/`map`이 더 빠르고, 시퀀스는 데이터가 클 때만 이득이다.

### 언제 함수를 인라인으로 선언할지 결정

`inline`을 무조건 붙이면 안 된다. 적절한 경우는 정해져 있다.

**인라인이 좋은 경우**
- 람다를 인자로 받는 함수: 람다 객체 생성 + invoke 호출 비용 제거
- `reified` 타입 파라미터를 쓰는 함수 (11장): 인라인이 아니면 reified 불가
- `return` 등 제어 흐름을 람다에서 활용해야 하는 경우 (비로컬 return)

**인라인을 피해야 하는 경우**
- 일반 함수: 본문이 그대로 복사되므로 바이트코드가 비대해진다. JIT 최적화도 못 받음
- 본문이 큰 함수: 호출 지점마다 코드가 복제되어 메서드 크기 폭발
- `public` API: 인라인 함수는 private 멤버에 접근할 수 없다. 변경 시 ABI 호환성 문제도 생김

경험칙: 람다 파라미터가 있고, 본문이 짧을 때만 `inline`을 붙인다.
Kotlin 표준 라이브러리도 이 원칙을 따른다.

### withLock, use, useLines로 자원 관리

`inline`의 또 다른 강점은 자원 관리 패턴(try-with-resources 같은)을 라이브러리 함수로 우아하게 표현할 수 있다는 점이다.

#### `withLock` — Lock 안전 사용

```kotlin
val l: Lock = ReentrantLock()
l.withLock {
    // 잠금이 걸린 상태에서 보호된 자원 접근
}
```

내부 구현:

```kotlin
inline fun <T> Lock.withLock(action: () -> T): T {
    lock()
    try {
        return action()
    } finally {
        unlock()
    }
}
```

#### `use` — Closeable 자원 자동 해제

Java의 try-with-resources에 해당. 모든 `Closeable`/`AutoCloseable`에서 사용 가능하다.

```kotlin
fun readFirstLineFromFile(path: String): String {
    BufferedReader(FileReader(path)).use { br ->
        return br.readLine()   // 비로컬 return 가능 (인라인이라서)
    }
    // 블록 종료 시 자동으로 close()
}
```

#### `useLines` — 파일 라인을 시퀀스로

```kotlin
File("data.txt").useLines { lines ->
    lines.filter { it.isNotBlank() }
         .forEach(::println)
}   // 블록을 벗어나면 파일이 닫힘
```

`lines`는 시퀀스라서 메모리 효율적이다. 다만 블록 밖으로 시퀀스를 꺼내면 안 된다(파일이 이미 닫혀 있음).

## 3. 람다에서 반환: 고차 함수에서 흐름 제어

람다 안에서 `return`을 쓰면 어디로 돌아갈까? 이것이 이 절의 핵심 질문이다.

### 람다 안의 return: 둘러싼 함수에서 반환

람다 안의 `return`은 람다 자체가 아니라, 람다를 둘러싼 함수(즉, 람다가 정의된 함수)에서 반환된다.
이를 **비로컬 return(non-local return)** 이라 한다.

```kotlin
data class Person(val name: String, val age: Int)

fun lookForAlice(people: List<Person>) {
    people.forEach {
        if (it.name == "Alice") {
            println("Found!")
            return                // lookForAlice 함수 자체에서 리턴
        }
    }
    println("Alice is not found")
}
```

이런 비로컬 return은 람다를 인자로 받는 함수가 `inline`일 때만 가능하다.
`forEach`는 인라인 함수라서 가능하다. 일반(non-inline) 고차 함수에서는 컴파일 에러가 난다.

### 레이블을 사용한 return

람다 자체에서만 빠져나오고 싶다면(즉, 로컬 return = `continue`처럼 동작), 레이블을 쓴다.

#### 명시적 레이블

```kotlin
fun lookForAlice(people: List<Person>) {
    people.forEach label@{
        if (it.name == "Alice") return@label   // 이 람다만 종료
    }
    println("Alice might be somewhere")   // 항상 실행됨
}
```

#### 함수 이름을 레이블로 사용

람다를 받는 함수의 이름을 레이블 대신 쓸 수 있다.

```kotlin
people.forEach {
    if (it.name == "Alice") return@forEach
    println(it.name)
}
```

같은 람다에 명시적 레이블과 함수 이름 레이블을 동시에 정의하면 명시적 레이블이 우선한다.

#### `this@레이블` — 수신 객체 지정 람다에서

`with`, `apply` 같은 함수에서는 레이블로 `this`를 구분할 수 있다.

```kotlin
println(StringBuilder().apply sb@{
    listOf(1, 2, 3).apply {
        this@sb.append(this.toString())   // sb는 바깥 StringBuilder
    }
})
```

### 익명 함수: 기본적으로 로컬 return

비로컬 return을 자주 쓰면 가독성이 떨어진다. 이때 **익명 함수(anonymous function)** 가 대안이 된다.

```kotlin
fun lookForAlice(people: List<Person>) {
    people.forEach(fun(person) {
        if (person.name == "Alice") return   // 익명 함수에서만 반환
        println("${person.name} is not Alice")
    })
}
```

#### 익명 함수의 특징

- `fun` 키워드로 작성 (이름 없음)
- 일반 함수처럼 반환 타입을 명시 가능 (식 본문이면 추론도 가능)
- `return`은 가장 가까운 `fun`으로 돌아간다 → 항상 로컬 return

```kotlin
people.filter(fun(person): Boolean {
    return person.age < 30
})

// 식 본문 형태
people.filter(fun(person) = person.age < 30)
```

#### 람다 vs 익명 함수의 return

| 형태 | `return`의 의미 |
|---|---|
| 람다 `{ ... }` | 비로컬 return (둘러싼 함수에서 반환) |
| 람다 + 레이블 `return@label` | 람다에서만 반환 |
| 익명 함수 `fun(...) { ... }` | 익명 함수에서만 반환 (로컬) |

두 형태 모두 함수 타입(`FunctionN`)의 값으로 컴파일된다. 차이는 `return`이 어디로 돌아가느냐 뿐이다.

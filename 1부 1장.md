1.2.1 p45, 1.3.3 p60, 1.5.2 p71)

2. Java 대비 어떤 문제를 해결하고, 어떻게 코드를 더 간결하게 만드는가? (김승수)

3. Kotlin이 함수형 프로그래밍을 지원하는 것이 왜 동시성 문제를 줄이는 데 도움이 되는가? (김규)
p49
    - first-class: callback, promise 와 같은 동시성 문제 처리 기법을 구현하는 데 유리
    - immutability: 동시성 환경에서 하나의 객체가 여러 주체에 의해 임의의 순서로 변경되는 것을 방지함
    - side effect: 부수효과를 없앤다는 컨셉이, 동시성 환경에서 여러 주체가 하나의 객체를 바꾸지 않음을 보장하여 결과물의 신뢰성을 높임
    1. 불변데이터는 Java도 지원하지 않나?
        1. 언어의 컨셉: 객체지향의 mutable state는 동시성 처리에 초점을 맞추지 않음
        2. 언어 자체의 기능: 자바스크립트처럼 구조적으로 불변성을 지원할 수 없는 언어도 있음

1. 정적타입 언어에서 성능 왜 빠른가?  (1.2.2 p.46) (김세진)
    - 어떤 메소드를 호출할지 **런타임에 알아볼 필요가 없다**는 무슨 의미일까? (동적 vs 정적 타입 언어 비교)
        - 정적타입 언어: 컴파일 시점에 바이트 코드에 작성됨, 런타임 시점에 메모리 주소를 알고 있기 때문에, 바로 사용할 수 있음.
        - 동적타입 언어: 런타임 시점에, 해당 메소드가 어디 있는 지 메모리 상에서 찾아야 함.
            - 파이썬,JS 는 동적 타입 언어, 코틀린 대비 컴파일이 빠르다는 장점은 있음. 하지만 런타임 시점에 오류를 만나면 디버깅하여 찾아야 함.

**(규)**

1. 람다 함수의 인라이닝은 어떻게 가비지 컬렉션의 동작으로 인한 멈춤 현상을 막을까? (김세진)
(p.59)
    - 💡람다 함수의 인라이닝은 객체를 생성하지 않는다. 함수를 인자로 받기 때문에, 객체를 만들지 않게 되고, 객체 증가로 인한 가비지 컬랙션 멈춤 현상을 막을 수 있다.
    - inline 키워드를 함수에 붙이게 되면, 컴파일러는 그 **함수를 호출하는 모든 문장을 함수 본문에 해당하는 바이트코드로 싹 바꿔치기해주기 때문에 오버헤드를 줄일 수 있다.**
    - **즉** 함수와 람다를 호출하는 대신, 해당 코드를 호출 위치에 직접 삽입해 객체 생성과 함수 호출 비용을 줄인다.
2. 코틀린과 자바의 차이가 무엇이길래 NPE를 코틀린은 컴파일 시점에 잡아낼 수 있고 자바는 잡아낼 수 없는 것일까? (김세진)
    - 자바는 자료형에 null을 구분하지 않고, null 이 모든 타입에 들어갈 수 있다.
    - 하지만 코틀린은 null을 명시하지 않으면, 그 값은 절대 null이 될 수 없고, 컴파일러가 null 안정성을 검사함.
        - 자바의 NPE 가 그렇게 많이 나냐 ⇒ YES. 예외 처리를 못한 휴먼 에러가 생각보다 빈번히 크리티컬하게 발생함.
3. 코틀린을 JVM에서 실행한다는 것은 어떤 장점을 가질까? (김세진)
(p.64)
    - 이미 성숙한 JVM 생태계와 성능 최적화를 그대로 활용할 수 있음. Java와 상호 운용성을 가지기 때문에, 기존 Java 프레임워크를 그대로 사용 가능.
        - 기존 JVM의 장점: 지금까지의 데이터, 메모리 안정성, 버퍼오버플로우 방지, 타입 안정성 보장 등…

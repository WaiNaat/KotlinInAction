# 9장. 연산자 오버로딩과 다른 관례

# 정리
- 관례
  - 어떤 언어 기능과 미리 정해진 이름의 함수를 연결해주는 기법
  - 기존 자바 클래스에 대해 확장함수를 구현하는 방식을 통해, 기존 자바 코드를 바꾸지 않고 새로운 기능을 쉽게 부여할 수 있음
- 산술 연산자
  - 코틀린에서 관례를 사용하는 가장 단순한 예
  - 연산자를 오버로딩할 떄는 `operator` 변경자 사용
  - 프로그래머가 직접 연산자를 만들 수 없음.
  - 언어에서 미리 정해둔 연산자만 오버로딩 가능.
    - `times`, `div`, `mod`, `plus`, `minus`
  - 연산자 우선순위는 항상 표준 숫자 타입의 연산자 우선순위와 같음
  - 두 피연산자가 항상 같은 타입일 필요는 없음
    - 교환법칙은 지원하지 않음 (오버로딩 각자 해서 구현은 가능)
  - 반환 타입이 반드시 두 피연산자 중 하나와 같을 필요도 없음
- 복합 대입 연산자
  - `plusAssign`: `+=`
  - 컴파일러는 `+=` 연산을 `plus` 또는 `plusAssign` 중 무엇을 선택해야 할지 모름. 따라서 둘 중 하나만 정의해야 함.
- 단항 산술 연산자
  - `unaryPlus`, `unaryMinus`, `not`, `int`, `dec`
  - `++a`, `a++` 와 같은 연산은 실행 순서만 다르지 내부 계산은 같으므로 별도로 정의할 필요 없음
- 동등성 연산자
  - `equals`: `==`
  - `===` 를 오버로딩할 수는 없음
  - `equals`는 `Any`에 이미 정의되어 있으므로 반드시 override 필요
    - 상속받은 함수가 확장 함수보다 우선순위가 높아서 `equals`는 확장함수로 만들 수 없음
- 순서 연산자
  - `Comparable` 인터페이스를 구현하면 부등식 사용 가능
  - `CompareValuesBy`
    - 두 객체와 여러 비교 함수를 인자로 받음
    - 처음 0이 아닌 값이 나오는 함수의 결과를 리턴
    - 간결한 코드로 비교 연산 가능
    - 필드를 직접 비교하는 게 성능상 좋긴 함
- 인덱스로 원소 접근
  - `get`, `set`
  - `set`이 받는 마지막 파라미터 값이 실제 값, 나머지는 인덱스 연산자에 들어감
- membership test
  - `in` 연산자: `contains` 함수
- 범위
  - `rangeTo`(닫힌구간), `rangeUntil`(왼쪽만 닫힌구간)
    - 다른 산술 연산자보다 우선순위가 낮음에 유의
  - `Comparable` 구현 시 정의할 필요 없음
- 루프
  - `for`
  - `Iterator<T>`를 구현하는 객체를 반환하는 `iterator` 메서드를 구현
- 구조 분해 선언
  - `componentN`
  - data class는 컴파일러가 자동으로 만들어줌
    - 필드 순서에 의존하기 때문에 위험
  - 함수에서 여러 값을 반환해야 하는데, 클래스를 정의하기 귀찮을 때 사용
    - 코드는 단순해지지만 표현력은 떨어짐
  - 크기가 정해진 컬렉션의 경우 구조 분해가 유용할 수 있음
  - `Pair`, `Triple`을 사용하는 게 일반적으로 더 깔끔함
- 위임
  - 객체가 직접 작업을 수행하지 않고 다른 도우미 객체가 그 작엄을 처리하도록 맡기는 디자인 패턴
- 위임 프로퍼티
  - 값을 backing field에 저장하는 것보다 더 복잡하게 동작하는 프로퍼티에 사용
  - 접근자 로직을 매번 재구현할 필요 없음
  - `by`
  - 도우미 객체는 다음 메서드를 제공해야 함
    - `getValue`
      - 외부에서 프로퍼티 접근 시 내부에서 이 함수 호출
    - `setValue`
      - 외부에서 프로퍼티 변경 시 내부에서 이 함수 호출
    - `provideDelegate`
      - 선택사항
      - 최초 생성 시 검증 로직 수행, 위임이 인스턴스화되는 방식 변경 등
      - 위임 객체를 생성할 때 이 함수가 구현되어 있으면 이것을 사용
  - `by` 오른쪽의 식이 반드시 새 인스턴스를 만들 필요는 없으나, 그 식의 계산 결과는 올바른 타입의 `getValue`와 `setValue`가 들어있어야 함.
  - 동작 방식
    - 도우미 객체는 별도의 숨겨진 프로퍼티에 저장됨
    - 컴파일러는 프로퍼티를 표현하기 위해 `KProperty` 타입의 객체를 사용함
    - 위임하는 원래 프로퍼티의 get과 set은 숨겨진 프로퍼티의 도우미 객체의 메서드를 `KProperty` 타입의 객체를 넘겨주는 방식으로 호출함
- 지연 초기화
  - 초기화 과정에 자원을 많이 사용할 경우
  - 객체를 사용할 때마다 꼭 초기화하지 않아도 되는 경우
  - `by lazy`를 사용하지 않고 backing property 를 만들어서 직접 구현하는 방법도 있으나, thread-safe 하지 않음.
  - thread-safe
- observable
  - 프로퍼티가 바뀔 때마다 리스너에게 변경 통지를 하는 기법
- 확장 가능한 객체
  - 자신의 프로퍼티를 동적으로 정의할 수 있는 객체를 만들 때 사용
  - `Map`은 `getValue`와 `setValue`를 확장함수로 제공함. 따라서 Map 자체를 도우미 객체로 활용할 수 있음. (이때 프로퍼티 이름이 map의 key가 됨)
- 프레임워크
  - 프레임워크에서 DB나 엔티티를 매핑하는 데 사용할 수 있음

### lazy 는 어떤 방식으로 thread-safe 한 것을 보장하는가?
- `lazy`의 모드
  - `SYNCHRONIZED`: lock으로 하나의 스레드만 초기화하도록 강제함
  - `PUBLICATION`: 제일 먼저 초기화한 놈이 이김
  - `NONE`: 아무 처리 하지 않음 (thread-safe 보장 X)
- 동작
  1. 프로퍼티 초기화 여부 확인
  2. 초기화 안 됐으면 lock 걸고 동기화 블록 진입
  3. 동기화 블록 내부에서 초기화 여부 다시 확인 (2번 과정을 동시에 통과한 스레드들이 있을까봐)
  4. 진짜 초기화

### `KProperty`란 무엇인가?
- 프로퍼티에 대한 메타정보를 담고 있는 객체
- 이름, 반환 타입, 어노테이션, 접근제한자 등등
- `::` 로 반환되는 게 바로 이거

### Q1) += 는 mutable 객체와 immutable 객체에서 어떻게 다르게 동작할까요?
- mutable
  - 객체 자체는 변하지 않고, 내부 값을 직접 바꿈
  - `plusAssign` 사용
- immutable
  - 연산 결과를 새로운 객체로 만든 후, 변수에 다시 할당
  - `var` 일 경우에만 사용 가능
  - `plus` 사용

- 둘 중 하나만 구현하거나, `val` 을 사용하기

### Q2) 동등성 비교(==)를 위한 equals 함수를 오버라이딩할 때, 왜 operator 키워드를 붙이지 않아도 될까요?
- `Any`에 이미 `operator fun equals`가 정의되어 있기 때문에 override 만 가능
- 상속받은 함수가 확장 함수보다 우선순위가 높아서 `equals`는 확장함수로도 만들 수 없음

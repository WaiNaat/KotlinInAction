# 정리
- 함수형 방식: 일반적인 연산을 일관성 있게 표현 가능
- `filter`, `filterIndexed`, `filterNot`
  - 컬렉션의 원소를 걸러냄
- `map`, `mapIndexed`
  - 컬렉션의 각 원소를 다른 형태로 변환함
- `reduce`
  - 컬렉션의 정보를 종합하는 데 사용
  - 원소로 이루어진 컬렉션을 받아 하나의 값을 반환
  - 컬렉션의 첫 값을 accumulator 에 넣음
- `fold`
  - 임의의 시작 값을 accumulator 에 넣음
  - 따라서 accumulator 의 타입도 지정 가능
- `runningReduce`, `runnungFold`
  - `reduce`, `fold` 의 중간 과정 값들을 전부 보여줌
- `all`
  - every
  - 빈 컬렉션에 대해서 true (vancuous truth)
- `any`
  - some
  - 빈 컬렉션에 대해서 false
- `none`
  - not some
  - 빈 컬렉션에 대해서 true
- `count`
  - 조건을 만족하는 원소의 개수를 반환
  - filter 후 size를 구하는 것과 비교했을 때, 배열을 생성하지 않고 숫자만 세므로 효율적
- `find`, `firstOrNull`
  - 조건을 만족하는 첫 번째 원소 반환
- `partition`
  - 이분법
  - 술어를 만족하는 첫 번쨰 그룹과 만족하지 않는 두 번째 그룹을 반환함
- `groupBy`
  - 컬렉션의 원소를 어떤 특성에 따라 나누고 싶을 때 사용
- `associate`
  - 원소를 그룹화하지 않으면서 컬렉션으로부터 맵을 만듦
  - key는 유일해야 하기 때문에 겹칠 경우 마지막 결과가 그 이전 결과를 덮어씀
  - 람다의 결과인 Pair 에서 first가 key, second 가 value 인 map 생성
- `associateWith`
  - 컬렉션의 원래 원소를 key로, 람다의 결과를 value로 사용
- `associateBy`
  - 람다의 결과를 key로, 원래 원소를 value로 사용
- `replaceAll`
  - `MutableList`에 적용 시 컬렉션의 모든 원소를 람다의 결과로 변경
- `fill`
  - 모든 원소를 같은 값으로 바꿈
- `ifEmpty`
  - 컬렉션에 아무 원소도 없을 때 기본값을 생성 가능
- `ifBlank`
  - 공백만 있는 문자열에 기본값을 지정 가능
- `windowed`
  - 슬라이딩 윈도우 생성
  - 원할 시 윈도우 출력의 결과를 변환하는 람다도 줄 수 있음
- `chunked`
  - 컬렉션을 주어진 크기의 서로소 집합으로 나눔
  - 원할 시 청크 출력의 결과를 변환하는 람다도 쓸 수 있음
  - 마지막 청크의 크기는 좀 작을 수도 있음
- `zip`
  - 두 컬렉션에서 같은 인덱스에 있는 원소들의 쌍으로 이루어진 리스트를 만듦
  - 람다로 출력 변환 가능
  - 필연적으로 두 컬렉션 중 짧은 쪽의 길이와 같은 길이의 배열이 반환됨
  - 중위 표기법으로 사용 가능하나 이 때 람다는 전달 불가
- `flatMap`
  - 컬렉션의 각 원소를 매핑한 후, 결과를 펼침
- `flatten`
  - 컬렉션을 펼침
- 시퀀스
  - 컬렉션 함수를 연쇄적으로 호출하면 매 단계마다 임시 컬렉션이 만들어짐
  - 시퀀스: 중간 임시 컬렉션을 만들지 않고 컬렉션 연산을 연쇄하는 방법
  - `Sequence` 인터페이스
    - 한 번에 하나씩 열거될 수 있는 원소의 시퀀스를 표현
    - `iterator` 라는 메서드 하나만 존재
  - 지연 계산
    - 결과 시퀀스의 원소를 하나씩 이터레이션하거나 리스트로 변환해야 진짜로 계산을 수행함
    - 원소 하나씩 계산하다가 결과가 얻어지면 이후 원소의 변환은 하지 않음
    - 큰 컬렉션에 대한 연쇄 연산에는 시퀀스 사용이 유리함
  - 시퀀스 연산의 종류
    - 중간 연산
      - 다른 시퀀스를 반환
      - 최초 시퀀스의 원소를 변환하는 방법을 앎
    - 최종 연산
      - 결과를 반환
  - 연쇄적인 연산에서 로직을 해치지 않는 선에서 filter 등을 먼저 해서 개수를 줄이면 성능에 좋음
  - `generateSequence` 로 시퀀스 직접 생성 가능
  - 그렇게 좋으면 시퀀스만 쓰면 되는거 아님?
    - 인덱스를 이용한 접근 등에는 사용할 수 없음


# 궁금했던 것들

### 빈 컬렉션에 `reduce`를 사용하지 말라고 한다. 설마 empty collection 전용 구현체가 있는 것이 이 때문인가?
- 아님
- 빈 컬렉션에 reduce는 `UnsupportedOperationException`
- 빈 컬렉션 구현체는 메모리 최적화를 위해 돌려쓰는 싱글톤

### `associateWith` 랑 `associateBy` 랑 너무 헷갈리는데 좋은 방법 없는지?
- gemini는 감동을 주지 못했음
- 그냥 ide에서 제공해주는 타입 보고 생각하기로 함

### `associate` 시리즈에서 key 가 겹치면 마지막이 이긴다고 했는데, 집합처럼 순서가 없는 컬렉션에서는 어떤 순서가 되는건지?
- 모든 컬렉션은 자체적으로 이터레이터를 갖고 있어서 그 순서를 따름
- 코틀린의 set은 기본적으로 linked hash set 이라 입력 순서를 따름
- 입력 순서를 보장하지 않는 HashSet 은 자체적인 계산법이 있음
  - (1) 버킷 순서 (2) 버킷 내 순서 
  - [JSK HashMap](https://github.com/openjdk/jdk/blob/master/src/java.base/share/classes/java/util/HashMap.java)

### `ifEmpty`, `ifBlank` 가 기본값을 '생성', '지정'한다고 하는데 원본을 변형시키는 것인지?
- 아님

### `windowed`, `chunked`, `zip` 은 순서가 있는 컬렉션에만 가능한 것인지?
- 아님
- `Iterable` 이면 다 가능함
- 하지만 순서를 예측할 수 없어서 명시적으로 순서가 있는 컬렉션에만 쓰는거임

### `flat`은 자바스크립트처럼 depth 지정이 가능한지?
- 코틀린은 컴파일 타임에 모든 타입이 알려져야 하므로 depth 를 정하면 계산이 힘들어서 무조건 하나만 까줌

### 시퀀스는 중간 연산을 어떤 방식으로 기억하는가?
- 데코레이터 패턴 사용
- 하나의 값에 대한 연산을 계속 감싸나가는 방식

### 시퀀스도 중간 객체를 만들어야 하는 것 아닌지? `map`만 연속으로 한다고 할 때, 시퀀스랑 배열이랑 차이가 있는건지?
- 중간 객체를 만들긴 하지만 대형 배열을 만드는 것보다는 저렴
- 중간 객체 1개를 이용한 연산이기 때문에 '배열의 순회' 관점에서 접근하면 횟수가 적음
- 데코레이터 적용 및 이터레이터 호출의 오버헤드가 있어서 짧은 배열은 시퀀스 안쓰는게 나을 수 있음

### 무한 시퀀스에 `toList()` 적용이 가능한지?
- 가능은 함
- 런타임 `OutOfMemoryError` 발생

### 코틀린은 내부적으로 무한 시퀀스와 유한 시퀀스를 구분하는지?
- iterator의 '끝'을 모르기 때문에 구분할 수 없음 (정지 문제)

### Q1) groupBy 와 associateBy 차이점에 대해 설명

- groupBy
  - 1대 N 매핑
  - 컬렉션의 원소를 어떤 특성에 따라 해당 특성을 갖는 친구들끼리 그룹화할 때 사용
- associateBy
  - 1대 1 매핑
  - 컬렉션에서 특정한 unique key를 기준으로 찾기 편하게 map을 만들고 싶을 떄 사용

### Q2) Sequence가 더 좋으면, 왜 Kotlin은 Sequence를 기본으로 안 만들었을까?

- sequence 는 iterator 만 가지고 있어서 인덱스 기반 접근이 불가능
- 시퀀스 연산에는 데코레이터 적용 및 이터레이터 호출의 오버헤드가 있어서 짧은 배열은 시퀀스 안쓰는게 나을 수 있음

### Q1) 빈 리스트(emptyList)에 .all { ... }를 수행하면 결과는 무엇일까요?

- true
- '모든 원소에 대해 참이다' = '거짓인 원소가 하나도 없다' 이므로 빈 배열에 대해 참
- 비슷한 예시
  - `any`: '해당하는 원소가 하나라도 있다'라서 애초에 원소가 없는 빈 컬렉션에 대해서 false
  - `none`: '해당하는 원소가 하나도 없다'라서 애초에 원소가 없는 빈 컬렉션에 대해서 true

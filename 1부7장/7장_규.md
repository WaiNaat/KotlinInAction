# 7장. 널이 될 수 있는 값

# 정리

- 타입
  - 가능한 값의 집합과 그 값들에 대해 수행할 수 있는 연산의 집합
- nullable type
  - 최신 언어에서는 컴파일 시점에 null을 판단하여 NPE를 예방하는 것이 트렌드
  - 코틀린 타입 시스템은 널이 될 수 있는 타입을 명시적으로 지원
  - 타입 이름 뒤에 `?` 를 붙여 타입의 변수나 프로퍼티에 null 참조를 저장할 수 있음
  - `?` 가 없는 타입은 어떤 변수가 null 참조를 저장할 수 없다는 의미
  - null 과 비교하여 smart cast 를 통해 뒤쪽 코드에서 편하게 사용 가능
- 자바
  - String 타입의 변수에는 String 또는 null 이 가능
  - 자바의 타입 시스템이 null을 제대로 다루지 못함
  - null 검사를 제대로 수행하지 않을 경우 NPE 발생
  - null 검사를 위한 어노테이션이 있으나 모든 NPE를 막기는 어려움
- 코틀린
  - 널이 될 수 있는 타입과 그렇지 않은 타입을 구분함
  - 실행 시점에는 두 타입을 구분하지 않음(객체가 같음)
- null 을 해결하는 새로운 방법
  - null 자체를 사용하지 않고, `Optional` 같은 wrapper type을 활용
  - 코드의 복잡성 증가, 성능에 영향 있음, 일관성 있는 활용 어려움
- `?.`
  - 안전한 호출 연산자
  - 호출하려는 값이 null이 아니면 일반 메서드 호출
  - null 이면 호출을 무시하고 null 반환
- `?:`
  - 엘비스 연산자
  - null 대신 사용할 기본값을 지정
  - return, throw 도 식이라서 편하게 사용 가능
- `as?`
  - 안전한 타입 캐스트 연산자
  - 어떤 값을 지정한 타입으로 변환. 불가능하면 null 반환.
- `!!`
  - 널 아님 단언
  - 어떤 값이든 not-null 로 강제로 바꿈
  - null에게 적용하면 NPE
    - 스택 트레이스에는 어떤 식에서 발생했는지의 정보가 없어서 한 줄에 여러 번 쓰면 추적이 어려울 수 있음
  - 컴파일러에게 이 값은 널이 아니며, 널일 때의 예외는 감수하겠다고 전달
  - 메서드 호출에 특별한 순서같은 것이 있어서 not-null 을 전달받는 것이 확실할 떄 사용 가능
- `?.let`
  - `let`을 안전한 호출 연산자와 사용해서 null 검사와 not-null일때의 처리를 편하게 가능
  - nullable 값을 not-nullable 만 받는 함수에 넘기는 경우
- 코틀린 영역 함수
  - `let`
    - 어떤 식의 결과를 변수에 담되 그 영역을 제한하고 싶을 때
  - `apply`
    - 빌더 스타일의 API를 사용할 때
  - `also`
    - 객체에 어떤 동작을 실행한 후, 원래의 객체를 다른 연산에 사용하고 싶을 때
  - `with`
    - 하나의 객체에 대해 이름을 반복하지 않으면서 여러 함수 호출을 한 세트로 묶고 싶을 때
  - `run`
    - 객체를 설정한 다음에 별도의 결과를 돌려주고 싶을 때
- `lateinit`
  - 클래스 안의 널이 아닌 프로퍼티는 반드시 생성자에서 초기화해야 함
  - `lateinit` 변경자를 사용해서 프로퍼티를 나중에 초기화 가능
  - `var` 에 대해서만 사용 가능
    - `val` 은 `final`로 컴파일되므로 지연 초기화 불가능
  - 초기화하기 전에 접근하면 `UnintializedPropertyAccessException`
    - 초기화하기 전에 접근하지 않는 것은 개발자의 책임
- 확장 함수
  - 널이 될 수 있는 타입에 대해 확장 함수를 정의하면 내부에서 null을 별도 처리 가능
- 타입 파라미터
  - 모든 타입 파라미터는 기본적으로 널이 될 수 있음
  - `?` 가 없어도 널이 될 수 있는 유일한 예외
- 코틀린의 플랫폼 타입
  - 코틀린이 널 관련 정보를 알 수 없는 타입
  - 컴파일러는 모든 연산을 허용하지만 NPE에 주의해야 함
  - 자바 API를 다룰 때는 조심할 것
  - 모든 자바 타입을 nullable 로 다루면 불필요한 null 검사가 들어갈 수 있어서 이렇게 사용함
  - 개발자가 직접 플랫폼 타입을 선언할 수는 없음
- 코틀린의 자바 상속
  - 코틀린에서 자바 메서드를 오버라이드할 때 nullable 인지 아닌지 개발자가 결정해야 함
  - 자바 클래스나 인터페이스를 코틀린에서 구현할 경우 널 가능성을 제대로 처리하는 것이 중요함

# 궁금했던 것들

### 애초에 자바는 왜 String 타입의 변수에 String 과 null 을 허용하게끔 만들었는가?\
- C언어의 포인터 개념이 적용됨 (포인터가 아무것도 가리키지 않는 경우에 대한 고려)

### 코틀린의 nullable type은 실행 시점 성능에 영향을 끼치는 몇 가지 근본적인 검사를 제외하면 부가 비용이 없다는데, 이게 부가 비용 아닌지? 얼마나 싸면 '비알콜 맥주'처럼 무시할 수 있는건지?
- `Optional` 과 비교하면 저렴한게 맞음
- 참조 타입은 실행 시점에는 nullable 과 아닌 것이 동일하므로 비용 차이가 없음
- primitive type 은 nullable 로 사용 시 boxing 이 적용되어 힙 메모리를 먹을 수 있긴 함

### 호출된 함수가 언제나 다른 함수에서 not-null 값을 전달받을 떄 `!!` 를 써서 불필요한 검사를 없앨 수 있다고 함. 하지만 이건 함수의 시그니처를 not-null 로 해야 하는 것이 아닌지? 애초에 순서 의존성이 있는 함수 구성을 짜는 것 자체가 잘못된 것은 아닌지?
- 일반적으로는 이게 맞음
- 자바 라이브러리랑 같이 쓰는 경우에는 별도 처리가 필요할 수 있음
- 프레임워크에서 동작을 보장하는 경우 써야할 수 있음

### 코틀린은 `lateinit`이 지정된 프로퍼티와 가시성이 똑같은 필드를 생성해준다고 함. 필드는 이미 있는 것이 아닌지? 이게 무슨 의미인지?
- 일반적인 `var`는 private backing field, getter, setter 세 종류로 분해해서 컴파일됨
- `lateinit`은 backing field 를 지정된 프로퍼티와 같은 가시성으로 만들어버린다는 뜻
- 프레임워크의 의존성 주입 호환성을 위해서 이렇게 함

### 컴파일러는 어떻게 안전한 호출 연산자 `?.` 와 nullable type 에 대한 확장함수 정의를 구분할 수 있을까?
- lexical analysis 단계에서는 같은 토큰으로 취급하지만 syntax analysis 단계에서 구분함
- 함수 선언부에서 만난 것인지를 판단

### `Any` 타입의 의미는 '널이 아닌 것'인가?
- ㅇㅇ
- `Any` 는 널이 아닌 객체라면 수행할 수 있어야 하는 `equals`, `hashCode`, `toString` 이 들어있음


## 7.1 NullPointerException을 피하고 값이 없는 경우 처리: 널 가능성

코틀린은 null 타입을 지원하면서 NPE를 예방한다. 널이 될 수 있는 타입을 살펴본다.

## 7.2 널이 될 수 있는 타입으로 널이 될 수 있는 변수 명시

코틀린은 널이 될 수 있는 프로퍼티나 변수의 메서드를 호출함으로써 NPE가 발생되게 하는 것을 막는다.

다음 자바 코드는 안전한가? String이 null이면 NPE가 발생한다.

```java
int strLent(String s){
	return s.length()
}
```

코틀린에서는 

```kotlin
// null이 될 수 없다
fun strLent(s: String)
	= s.length()
	
// null이 가능하다
fun strLent(s: String?):Int
	= s?.length() ?: 0
```

타입 뒤에 ? 가 붙는다면 그 타입은 null이 될 수 있다는 것을 의미한다.

널이 될 수 있는 값을 널이 될 수 없는 타입의 변수에 대입할 수 없다.

```kotlin
fun main() {
	val x: String? = null
	var y: String = x
	// ERROR: Type mismatch
}
```

그렇다면 null이 될 수 있는 타입과 비교한 후에는? 컴파일러가 null이 아님을 확실히 한다면 컴파일이 된다.

```kotlin
fun strLengSafe(s: String?): Int = 
	if(s != null) s.length else 0
```

## 7.4 안전한 호출 연산자로 null 검사와 메서드 호출 합치기: ?.

코틀린이 제공하는 안전한 호출 연산자 ?.이 잇다. ?. 은 null검사와 메서드 호출을 한 연산으로 수행한다

ex) str?.uppercase()는 if(str ≠ null) str.uppercase() else null과 같다.

호출하려는 값이 null이면 null이 리턴되고 아니면 결과값이 리턴된다.

메서드 호출뿐 아니라 프로퍼티를 읽거나 쓸 때도 안전한 호출을 사용할 수 있다.

```kotlin
class Employee(val name: String, val manager: Employee?)

fun managerName(employee: Employee): String? = employee.manager?.name
```

객체 그래프에서 널이 될 수 있는 중각 객체가 여럿 있따면 한 식 안에서 안전한 호출을 연쇄해서 함께 사용하면 편할 때가 자주 있따.

```kotlin
class Address(val streetAddress: String, val zipCode: Int,
 val city: String, val country: String)

class Company(val name: String, val address: Adderess?)

class Person(val name: String, val company: Company?)

fun Person.countryName(): String 
	= company?.address?.streetAddress ?: "Unknown"
```

## 7.5 엘비스 연산자로 null에 대한 기본값 제공: ?:

코틀린은 null 대신 사용할 기본값을 지정할 때 편리하게 사용할 수 있는 **엘비스 elvis** 연산자를 제공한다.

```kotlin
fun greet(name: String?){
	val recipient: String = name?: "unnamed"
	println("Hello, $recipient")
}
```

null이 아니면 그 값을 null이면 unnamed를 출력한다.

코틀린에서 return 이나 throw 등도 식이기 때문에 엘비스 연산자의 오른쪽에 return, throw등을 넣을 수 있다.

```kotlin
fun printShippingLabel(person: Person){
	val address = person.campany?.address ?: throw IllegalArgumentException("No Add")
	
	with(address){
		println(streetAddress)
		println("$zipCOode $city, $country")
	}
}
```

## 7.6 예외를 발생시키지 않고 안전하게 타입을 캐스트하기: as?

2장에서 코틀린 타입 캐스트 연산자인 as를 살펴본다. 자바 타입 캐스트와 마찬가지로 대상 값을 as로 지정한 타입으로 바꿀 수 없으면 ClassCastException이 발생한다. 이 때 as? 연산자를 사용해 변환할 수 없으면 null이 반환되게 할 수 있다.

equals를 구현할 때 유용하다

```kotlin
class Person(val firstName: String, val lastName: String){
	override fun equals(o: Any?): Boolean {
		val otherPerson = o as? Person ?: return false
		
		return otherPerson.firstName == firstName &&
			otherPerson.lastName == lastName
	}
}
```

값이 원하는 타입인지 검사하면 아니면 false를 반환하게 할 수 있다.

## 7.7 널 아님 단언: !!

널 아님 단언은 코틀린에서 널이 될 수 있는 타입의 값을 다룰 때 사용할 수 있는 도구중 가장 단순하면서 무딘 도구이다. 느낌표를 이중(!!)으로 사용하면서 어떤 값이든 널이 아닌 타입으로 바꿀 수 있다.

```kotlin
fun ignoreNulls(str: String?){
	val strNotNull: String = str!!
	println(strNotNull.length)
}
```

액션 클래스에서 널 아님 단언문 사용하기

```kotlin
class SelectableTextList(
	val contents: List<String>,
	var selectedIndex: Int? = null
)

class CopyRowAction(val list: SelectableTextList){
	fun isActionEnabled(): Boolean = 
		list.selectedIndex != null
	
	fun executeCopyRow(){
		val index = list.selectedIndex!!
		val value = list.contents[index]
	}
}
```

## 7.8 let 함수

let 함수를 사용하면 널이 될 수 있는 식을 더 쉽게 다룰 수 있다. 

let 함수를 안전한 호출 연산자와 함께 사용하면 원하는 식을 평가해서 결과가 null인지 검사한 다음에 그 결과를 변수에 넣는 작업을 간단한 식을 사용해 한꺼번에 처리할 수 있다.

```kotlin
fun sendEmailTo(email: String){/** **/}

fun main() {
	val email: String? = "foo@bar.com"
	sendEmailTo(email)
	// ERROR: Type mismatch: inferred type is String? but String was expected
	
	// 인자를 넘기기 전에 null인지 검사해야한다
	if(email != null) sendEmailTo(email)
	
	//let을 사용하면
	email?.let { email -> sendEmailTo(email) }
}
```

코틀린의 영역 함수 비교: with, apply, let, run, also 언제 사용할까?

| 함수 | x를 어떤 방식으로 참조하는가? | 반환값 |
| --- | --- | --- |
| x.let {…} | it | 람다의 결과 |
| x.also {…} | it | x |
| x.apply {…} | this | x |
| x.run {…} | this | 람다의 결과 |
| with(x) {…} | this | 람다 결과 |

let : null이 아닌 경우에만 코드 블록을 실행하고 싶으면 let을 안전한 호출 연산자 ?.와 함께 사용하라. 어떤 식의 결과를 변수에 담되 그 영역을 한정시키고 싶을 때는 let을 독립적으로 사용하라

apply : 빌더 스타일의 API를 사용해 객체 프로퍼티를 설정할 때

also : 객체에 어떤 동작을 실행한 후 원래의 객체를 다른 연산에 사용하고 싶을 때

with : 하나의 객체에 대해 이름을 반복하지 않으면서 여러 함수 호출을 그룹으로 묶고 싶을 때

run : 객체를 설정한 다음에 별도의 결과를 돌려주고 싶을 때 run 을 사용하라

## 7.9 직접 초기화하지 않는 널이 아닌 타입: 지연 초기화 프로퍼티

객체를 일단 생성한 다음에 나중에 전용 메서드를 통해 초기화를 하는 프레임워크가 있다. 안드로이드에서는 onCreate에서 액티비티를 초기화한다. JUnit에서는 @BeforeAll이나 @BeforeEach로 어노테이션된 메서드 안에서 초기화 로직을 수행한다.

코틀린에서는 생성자 안에서 널이 아닌 값들은 초기화되어야 한다. 이를 해결하기 지연 초기화 lateinit변경자를 지원한다.

```kotlin
class MyService{
	fun performAction(): String = "Action Done!"
	
	@TestInstance(TestInstance.Lifecycle.PER_CLASS)
	class MyTest {
		private lateinit var myService: MyService
		
		@BeforeAll fun setUp() {
			myService = MyService()
		}
		
		@Test fun testAction(){
			assertEquals("Action Done!", myService.performAction())
		}
	}
}
```

생성자 밖에서 값을 바꿔야 하는데, val 프로퍼티는 파이널 필드로 컴파일되며 생성자 안에서 반드시 초기화돼야 하기 때문에 **지연 초기화 프로퍼티는 항상 var**여야 한다는 점에 유의하자.

초기화 하기 전에 프로퍼티에 접근하면 다음과 같은 오류를 볼 수 있다.

kotlin.UninitializedPropertyAccessException: lateinit property myService has not been initialized

예외를 보면 어디서 잘못됐는지 확실히 알 수 있어서 단순한 NullPointerException이 발생하는 것보다 훨씬 좋다.

lateinit 프로퍼티를 구글 주스 등의 자바 의존관계 주입 프로임워크와 함께 사용할 수도 있다. lateinit 프로퍼티의 값을 DI프레임워크가 외부에서 설정해준다. 

다양한 자바 프레임워크와 호환을 위해 lateinit으로 지정된 프로퍼티와 가시성이 똑가은 필드를 생성해준다.
지연 초기화 프로퍼티가 public 이라면 코틀린이 생성한 필드도 public 이다.

lateinit 프로퍼티가 반드시 클래스의 멤버일 필요는 없다. 함수 본문 안의 지역 변수나 최상위 프로퍼티도 지연 초기화할 수 있다.

## 7.10 안전한 호출 연산자 없이 타입 확장: 널이 될 수 있는 타입에 대한 확장

```kotlin
fun verifyUserInput(input: String?) {
	if (input.isNullOrBlank()){ // 안전한 호출이 필요 없다.
		println("Plz fill in the required fields")
	}
}
```

안전한 호출 없이도 널이 될 수 있는 수신 객체 타입에 대해 선언된 확장 함수를 호출 가능하다. 

확장함수가 null을 잘 처리한다.

```kotlin
fun sendEmailTo(eamil: String) {
	println("Sending email to $email")
}

fun main() {
	val recipient: String? = null
	recipient.let { sendEmailTo(it) }
	// ERROR: Type mismatch
	recipient?.let { sendEmailTo(it) }
}
```

## 7.11 타입 파라미터의 널 가능성

코틀린에서 함수, 클래스 등 모두 null이 될 수 있다. 널이 될 수 있는 타입을 포함하는 어떤 타입이라도 타입 파라미터를 대신할 수 있다.

```kotlin
fun <T> printHashCode(t: T) { // 물음표가 없어도 널이 될 수 있는 타입이다
	println(t?.hashCode()) // t가 널이 될 수 있으므로 안전한 호출을 써야만 한다.
}

fun <T: Any> printHashCode(t: T) { // 이제 T는 null이 될 수 없다.
	println(t.hashCode()) // t가 널이 될 수 없으므로 안전하다.
}
```

## 7.12.1 플랫폼 타입

플랫폼 타입은 코틀린이 널 관련 정보를 알 수 없는 타입을 말한다. 널이 될 수 있는 타입으로 처리해도 되고 될 수 없는 타입으로 처리해도 된다. 책임이 코드를 짜는 사람에게 있다는 뜻이다. (컴파일러가 허용한다.)

플랫폼 타입 값이 널이 아님을 알고 있다면 아무 null 검사 없이 그 값을 직접 사용해도 된다.

만약 틀렸다면 NPE가 발생한다.

```java
public class Person {
	private final String name;
	
	public Person(String name){
		this.name = name;
	}

	public String getName(){
		return this.name;
	}
}
```

코틀린은 null인지 아닌지 알지 못한다. 우리가 직접 처리해야 한다.

```kotlin
fun yellAt(person: Person){
	println(person.name.uppercase() + "!!!") // 만약 null이면 NPE발생
}

fun yellAt(person: Person){
	println((person.name ?: "AnyOne").uppercase() + "!!!") // nullable로 처리
}
```

<aside>
💡

**코틀린은 왜 플랫폼 타입을 도입했을까?**

모든 타입이 널이 될 수 있다고 처리하면 손해볼 것이 더 많기 때문이다.

불필요한 null 검사가 많아지며, ArrayList<Strgin>를 코틀린에서 ArrayList<String>?로 다루면 배열의 원소에 접근할 때마다 null 검사를 수행하거나 안전한 캐스트를 수행해야한다.

</aside>

코틀린에서 플랫폼 타입을 선언할 수 없다. 자바 코드에서 가져온 타입만 플랫폼 타입이 된다. 하지만 IDE나 컴파일 오류 메시지에서는 플랫폼 타입을 볼 수도 있다.

val i: Int = person.name

// ERROR: Type mimatch: inferred type is Strin! but Int was expected

**String!** 은 코틀린 컴파일러와 IDE에서 자바 코드에서 온 플랫폼 타입을 표시하는 방법이다.

이 표기는 String! 타입의 널 가능성에 대해 아무 정보도 없다는 사실을 강조할 뿐이다.

val s: String? = person.name

val s1: String = [person.name](http://person.name)  둘 다 컴파일은 이상이 없다.

## 7.12.2 상속

코틀린에서 자바 메서드를 오버라이드 할 때 그 메서드의 파라미터와 반환 타입이 널이 될 수 있는 타입으로 선언할지 널이 될 수 없는 타입으로 선언할지 결정해야 한다.

```java
interface StringProcessor {
	void process(String value);
}
```

코틀린 컴파일러는 다음과 같은 두 구현을 받아들인다.

```kotlin
class StringPrinter: StringProcessor {
	override fun process(value: String) {
		println(value)
	}
} 

class NullableStringPrinter: StringProcessor {
	override fun process(value: String?){
		if(value != null){
			println(value)
		}
	}
}
```

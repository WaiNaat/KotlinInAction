### Q1) 기능상 singleton 과 top-level property/function 이 할 수 있는 일이 비슷한데, 둘을 구분해서 사용할만한 이유나 상황이 있을까요?

**A) 김세진**

(p.138, p.218)

* 둘 다 전역접근이 가능하지만, 역할과 설계 목적에 따라 나뉨.

  * top-level은 단순 static 유틸 함수에 적합하고
  * singleton는 상태 관리나 인터페이스 구현 등 객체 개념이 필요할 때 사용합니다

* top-level property/function

  * 확장함수 사용할 때
  * 상태가 없고, 단순 유틸리티 계산(입력값에 따라 결과만 내놓는 함수) 등의 함수일 때

```kotlin
fun validateUser(name: String){}

val PI = 3.14

//JVM
public final class FileKt {

    public static void validateUser(){}

}
```

* singleton

  * 상태가 필요하고
  * 인터페이스 구현이 필요 할 때
  * 관련 기능을 묶어야 할 때

    * top-level 은 불필요한 레퍼클래스(중첩 구조) 를 없애고자 만든 것이므로,
  * 여러 프로퍼티의 상태를 하나의 단위로 묶어서 관리해야 할 때

```kotlin
object UserValidator {
    fun validate(name:String){}
}
```

### Q3) 아래 코드에서, customer1 == customer2 가 제대로 작동하지 않는 경우가 있다. 문제가 무엇인가

**A) 김규**

p209 질문

```kotlin
class Customer(val name: String, val postalCode: Int){
	override fun equals(others: Any?):Boolean {
		if (other == null || other !is Customer)
			return false
		return name == other.name && postalCode == other.postalCode
		}
}

val customer1 = Customer("Alice", 123523)
val customer2 = Customer("Alice", 123523)
```

* HashSet(HashMap) 에 들어있는 경우 문제가 될 수 있음
* hashCode 메서드가 없다.

  * JVM 언어에서는 equals 가 true 를 반환하는 두 객체는 같은 hashCode 를 반환해야 함
  * hashSet 은 비용 절감을 위해 hashCode() 가 같은 경우에만 실제 값을 비교
  * 메서드를 제대로 구현하지 않았으므로 위쪽 규칙이 지켜진다는 보장이 없다

### Q6) 두 인터페이스 A, B가 동일한 시그니처의 디폴트 메서드 `showOff()`를 가질 때, 이를 구현하는 클래스는?

**A) 김세진**

* 동일한 메서드가 있을 때에는, 해당 클래스에서 그 메소드를 오버라이드해서 구현해야 하고,
  특정 부모의 기능을 호출하고 싶다면 super<T> 구문을 사용하여 부모 호출

```kotlin
override fun showOff() {
 super<A>.showOff()
 super<B>.showOff()
}
```

* 하지만! 이렇게 사용하는 것 보다 메소드명을 바꾸는 게 더 나은듯

### Q7) 자바에서 코틀린 companion object의 메서드를 `MyClass.method()`처럼 정적 호출하려면?

**A) 김규**

* 동반 객체는 정적 필드로 컴파일된다.
* `{클래스명}.{동반객체명}` 으로 호출할 수 있으며, 동반 객체의 이름 기본값은 `Companion` 이다.
* `@JvmStatic`, `@JvmField` annotation으로 동반 객체의 이름 없이 사용 가능하다.

### Q8) `@JvmInline value class UsdCent(val amount: Int)`의 런타임 동작은?

**A) 김세진**

**필요한 경우(nullable, generic 등)에는 boxing되어 객체로 동작합니다.**

* **컴파일 타임에는 별도 타입으로 검사하고, 런타임에서는 내부 값으로 대체된다.**

* 즉 **Type safety는 유지하면서 성능 오버헤드는 제거하는 구조**

* 컴파일 후 JVM: `voidaddExpense(intexpense)`

  * **UsdCent 객체가 생성되지 않고 Int로 대체**

    * == inline representation 또는 unboxed representation

      * 타입 구분은 컴파일 타임에만 존재
      * 근데, 타입이 Any 거나, Generic, nullable 등으로 생성하게 되면, Boxing 이 필요해서 객체를 생성하게 된다

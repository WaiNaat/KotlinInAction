
# 11장 제네릭스

## 1. 타입 인자를 받는 타입 만들기: 제네릭 타입 파라미터

제네릭스(generics)는 **타입을 파라미터화**하는 기능이다. 클래스나 함수가 다루는 데이터의 타입을 사용 시점에 결정할 수 있게 한다.

```kotlin
val list: List<String> = listOf("a", "b", "c")   // 타입 인자: String
val map: Map<String, Int> = mapOf("a" to 1)      // 타입 인자: String, Int
```

코틀린의 제네릭스는 자바와 호환되면서도,**선언 지점/사용 지점 변성**과 **실체화된 타입 파라미터(reified)** 을 지원

### 제네릭 함수와 프로퍼티

제네릭 함수는 함수 선언에 **타입 파라미터**를 추가한다.
```kotlin
fun <T> List<T>.slice(indices: IntRange): List<T> {
    return indices.map { this[it] }
}
```

```kotlin
val letters = ('a'..'z').toList()
letters.slice<Char>(0..2)   // 명시적
letters.slice(0..2)         // 컴파일러가 추론 (T = Char)
```

표준 라이브러리의 `filter`, `map`, `flatMap` 등 거의 모든 컬렉션 함수가 제네릭 함수다!!

#### 제네릭 확장 프로퍼티

확장 프로퍼티도 타입 파라미터를 가질 수 있다.

```kotlin
val <T> List<T>.penultimate: T
    get() = this[size - 2]

listOf(1, 2, 3, 4).penultimate   // 3
```

**일반 프로퍼티(확장이 아닌)는 제네릭이 될 수 없다.** 클래스 인스턴스 필드 하나에 타입이 다른 값을 저장할 수 없기 때문이다.

```kotlin
val <T> x: T = TODO()   // 컴파일 에러
```

### 제네릭 클래스 선언

클래스 이름 뒤에 홑화살괄호로 타입 파라미터를 선언한다.

```kotlin
interface List<T> {
    operator fun get(index: Int): T
}
```

상속받을 때는 타입 인자를 지정한다.

```kotlin
class StringList : List<String> {
    override fun get(index: Int): String = ...
}

class ArrayList<T> : List<T> {
    override fun get(index: Int): T = ...
}
```

자기 자신을 타입 인자로 참조하는 패턴(F-bounded polymorphism)도 가능하다.

```kotlin
interface Comparable<T> {
    fun compareTo(other: T): Int
}

class String : Comparable<String> {
    override fun compareTo(other: String): Int = ...
}
```

### 타입 파라미터 제약

타입 파라미터가 만족해야 하는 **상한(upper bound)** 을 지정할 수 있다. 콜론(`:`) 뒤에 상한 타입을 적는다.

```kotlin
fun <T : Number> List<T>.sum(): T = ...

listOf(1, 2, 3).sum()         // OK (Int : Number)
listOf("a", "b").sum()        // 컴파일 에러 (String은 Number가 아님)
```

상한이 지정되면 함수 본문에서 그 타입의 메서드를 호출할 수 있다.

```kotlin
fun <T : Comparable<T>> max(a: T, b: T): T {
    return if (a.compareTo(b) > 0) a else b
}
```

#### 여러 상한 지정: `where` 절

타입 파라미터에 두 개 이상의 제약이 필요하면 `where` 절을 사용한다.

```kotlin
fun <T> ensureTrailingPeriod(seq: T)
    where T : CharSequence, T : Appendable {
    if (!seq.endsWith('.')) {
        seq.append('.')
    }
}
```

`T`는 `CharSequence`이면서 `Appendable`이어야 한다. 두 인터페이스의 메서드를 모두 호출할 수 있다.

### 널이 될 수 없는 타입 파라미터

타입 파라미터에 상한을 지정하지 않으면 기본 상한은 **`Any?`** 다. 즉 `T`는 nullable일 수도 있다.

```kotlin
class Processor<T> {
    fun process(value: T) {
        value?.hashCode()   // T가 nullable일 수 있으므로 ?. 필요
    }
}

Processor<String?>()   // OK
```

`T`를 **non-null로 강제**하려면 상한을 `Any`로 지정한다.

```kotlin
class Processor<T : Any> {
    fun process(value: T) {
        value.hashCode()   // T가 non-null이므로 안전
    }
}

Processor<String?>()   // 컴파일 에러
Processor<String>()    // OK
```

이전 스터디에서 다뤘던 내용인데 이렇게 나왔다!

## 2. 실행 시점 제네릭스 동작: 소거된 타입 파라미터와 실체화된 타입 파라미터

JVM의 제네릭스는 **타입 소거(type erasure)** 방식이다.
컴파일 시점에는 타입 정보가 사용되지만, 런타임에는 모두 지워진다.
코틀린도 이 방식을 따르지만 **`inline` 함수와 `reified` 키워드**를 결합하면 일부 상황에서 이 한계를 우회할 수 있다.

### 실행 시점의 제네릭: 타입 검사와 캐스팅

#### 타입 소거의 결과

```kotlin
val list1: List<String> = listOf("a", "b")
val list2: List<Int> = listOf(1, 2)
```

이 두 리스트는 컴파일 후 모두 `List` 타입이 된다. `<String>`, `<Int>` 정보는 사라진다.
런타임에 타입 인자를 확인할 수 없다.

```kotlin
if (value is List<String>)   // 컴파일 에러
```

코틀린은 이를 컴파일 단계에서 막는다. 대신 **스타 프로젝션**으로 "어떤 타입 인자든 상관없는 리스트"인지는 검사할 수 있다.

```kotlin
if (value is List<*>) { ... }   // OK
```

`as` 캐스팅은 컴파일 경고가 뜬다(unchecked cast). 런타임에 타입 인자를 검사할 수 없으므로 캐스팅이 실제로 안전한지 알 수 없기 때문이다.

```kotlin
fun printSum(c: Collection<*>) {
    val intList = c as? List<Int>          // 경고: unchecked cast
        ?: throw IllegalArgumentException()
    println(intList.sum())
}

printSum(listOf("a", "b"))   // ClassCastException 대신 런타임 에러
```

`List<Int>`인지 검사하는 시점은 사실 `List`인지만 검사하는 것이다. 잘못된 타입의 리스트가 들어와도 캐스팅 자체는 성공하고, 나중에 요소를 사용할 때 비로소 실패한다.

#### 타입 소거의 이득

타입 소거가 단점만 있는 건 아니다.

- **메모리 절약**: 인스턴스마다 타입 정보를 보관할 필요 없음
- **자바 호환성**: 자바와 같은 방식이라 상호운용이 자연스러움

### 실체화된 타입 파라미터

`inline` 함수에 한해 타입 파라미터를 **실체화(reify)** 할 수 있다. `reified` 키워드를 붙이면 함수 본문에서 그 타입을 일반 클래스처럼 사용할 수 있다.

```kotlin
inline fun <reified T> isA(value: Any) = value is T

isA<String>("abc")    // true
isA<String>(123)      // false
```

10장에서 본 `inline`의 메커니즘 덕분에 가능하다. 인라인 함수 본문이 호출 지점에 펼쳐질 때, **`T`가 실제 타입(예: `String`)으로 치환**된다. 컴파일 결과는 다음과 비슷하다.

```kotlin
// 호출 지점에서 펼쳐진 모습:
"abc" is String   // T가 String으로 치환됨
```

#### `filterIsInstance`로 실용성 확인

표준 라이브러리에서 `reified`를 가장 잘 활용한 예다.

```kotlin
public inline fun <reified R> Iterable<*>.filterIsInstance(): List<R> {
    val destination = ArrayList<R>()
    for (element in this) {
        if (element is R) destination.add(element)   // R이 살아있음
    }
    return destination
}

val items: List<Any> = listOf(1, "two", 3.0, "four")
items.filterIsInstance<String>()   // ["two", "four"]
```

### 클래스 참조 대신 reified

자바에서는 런타임 타입 정보가 필요할 때 `Class<T>` 파라미터를 넘기는 패턴을 자주 사용한다.

```java
// Java 스타일
<T> T loadService(Class<T> serviceClass) { ... }

loadService(MyService.class);
```

코틀린에서는 `reified`로 더 깔끔하게 표현할 수 있다.

```kotlin
inline fun <reified T> loadService(): T {
    return ServiceLoader.load(T::class.java).first()
}

val service = loadService<MyService>()
```

`T::class.java`로 `Class` 객체에 접근할 수 있다는 점이 핵심이다. 일반 제네릭 함수에서는 불가능하다 (`T`가 런타임에 존재하지 않으므로).

안드로이드/리트로핏 등의 라이브러리에서 자주 보는 패턴.

```kotlin
// Retrofit 스타일 헬퍼
inline fun <reified T> Retrofit.create(): T = create(T::class.java)

// 안드로이드 startActivity
inline fun <reified T : Activity> Context.startActivity() {
    startActivity(Intent(this, T::class.java))
}

startActivity<DetailActivity>()
```

자바 API의 `Class<T>` 파라미터를 호출자에게서 숨길 수 있다.

### 실체화된 타입 파라미터의 제약

`reified`는 강력하지만 제약이 많다.

- **`inline` 함수에서만 사용 가능.** 인라이닝이 없으면 타입 정보를 보존할 방법이 없다.
- **클래스나 일반(non-inline) 함수의 타입 파라미터는 reified 불가.**
- **타입 파라미터로 인스턴스를 생성할 수 없다.** `T()` 같은 호출은 안 됨 (생성자 정보가 없음).
- **타입 파라미터의 동반 객체 메서드 호출 불가.** `T.companionMethod()` 같은 것도 안 됨.
- **자바에서 reified 함수를 일반 함수로는 호출 불가.** 코틀린에서만 호출하거나, 자바에서는 명시적으로 `Class` 받는 오버로드를 제공해야 한다.

이런 제약은 모두 같은 원인에서 나온다: `reified`는 컴파일 시점에 타입을 알아야만 작동한다는 것.

### 실체화된 타입 파라미터로 어댑터 정의하기

`reified`의 실용적 응용 예 하나. 안드로이드의 `findViewById`나 JSON 파싱 라이브러리 호출이 `reified` 덕분에 자연스러워진다.

```kotlin
// 일반적인 JSON 파싱 (Class를 받음)
val user: User = gson.fromJson(json, User::class.java)

// reified로 감싼 헬퍼
inline fun <reified T> Gson.fromJson(json: String): T = 
    fromJson(json, T::class.java)

val user: User = gson.fromJson(json)   // 타입 추론 + 캡슐화
```

이 패턴이 코틀린 라이브러리 곳곳에서 사용된다. 자바 라이브러리를 코틀린답게 감싸는 가장 흔한 방식.

## 3. 변성: 제네릭과 타입 인자 사이의 하위 타입 관계

**변성(variance)** 은 제네릭 타입이 타입 인자의 하위 타입 관계를 어떻게 보존(또는 뒤집기, 또는 무시)하는지를 정의한다. 코틀린의 변성 시스템은 자바보다 정교하고 명시적이다.

### 변성이 있는 이유: 함수에 인자 넘기기

다음 코드를 보자.

```kotlin
fun printContents(list: List<Any>) {
    println(list.joinToString())
}

val strings = listOf("a", "b")
printContents(strings)   // List<String>을 List<Any> 자리에 넘김
```

`String`이 `Any`의 하위 타입이므로 `List<String>`을 `List<Any>` 자리에 넘기는 게 직관적으로는 안전해 보인다. 그리고 실제로 코틀린은 이를 허용한다.

하지만 `MutableList`라면?

```kotlin
fun addAnswer(list: MutableList<Any>) {
    list.add(42)
}

val strings = mutableListOf("a", "b")
addAnswer(strings)   // 컴파일 에러
```

만약 이게 허용되면 `MutableList<String>`에 `Int`가 들어가는 사태가 벌어진다. 코틀린은 이를 컴파일 단계에서 막는다.

**핵심**: `List`(읽기 전용)는 타입 인자 관계가 보존되어야 하고, `MutableList`(읽기+쓰기)는 보존되면 안 된다. 이를 표현하는 메커니즘이 변성이다.

### 클래스, 타입, 하위 타입

용어 정리.

- **클래스(class)**: `String`, `List` 같은 선언.
- **타입(type)**: `String`, `String?`, `List<String>`, `List<String?>` 등 클래스에 타입 인자와 nullability가 결합된 결과.
- **하위 타입(subtype)**: 타입 A 자리에 타입 B를 사용해도 문제가 없으면 B는 A의 하위 타입.

`Int`는 `Number`의 하위 타입이다. 그럼 `List<Int>`는 `List<Number>`의 하위 타입일까? 이게 변성이 답하는 질문이다.

### 공변성(covariance): 하위 타입 관계 유지

타입 파라미터에 `out`을 붙이면 **공변(covariant)** 이 된다. 즉, `A`가 `B`의 하위 타입이면 `Producer<A>`도 `Producer<B>`의 하위 타입이다.

```kotlin
interface Producer<out T> {
    fun produce(): T
}
```

이제 `Producer<Cat>`은 `Producer<Animal>`의 하위 타입이다.

```kotlin
class CatProducer : Producer<Cat> { ... }

val animalProducer: Producer<Animal> = CatProducer()   // OK
```

#### `out`의 제약

`out` 타입 파라미터는 **반환 위치(out-position)에서만** 사용 가능하다. 파라미터 타입(in-position)으로는 쓸 수 없다.

```kotlin
interface Producer<out T> {
    fun produce(): T          // OK (out 위치)
    fun consume(value: T)     // 컴파일 에러 (in 위치)
}
```

왜? 만약 `consume(value: T)`가 허용되면 `Producer<Animal>.consume(Dog())`이 가능해지는데, 실제 구현체가 `Producer<Cat>`이면 `Cat.consume(Dog())`이 되어버려 타입 안전성이 깨진다.

**`List<T>`가 `out T`로 선언된 이유**: 읽기 전용 컬렉션이므로 `T`는 반환에만 쓰인다 (`get`, `iterator` 등). 따라서 공변이 안전하다.

### 반공변성(contravariance): 하위 타입 관계 뒤집기

타입 파라미터에 `in`을 붙이면 **반공변(contravariant)** 이 된다. `A`가 `B`의 하위 타입이면 `Consumer<B>`가 `Consumer<A>`의 하위 타입이다 (방향이 뒤집힘).

```kotlin
interface Consumer<in T> {
    fun consume(value: T)
}
```

```kotlin
val animalConsumer: Consumer<Animal> = ...

val catConsumer: Consumer<Cat> = animalConsumer   // OK
// Animal을 받을 수 있는 컨슈머는 Cat도 당연히 받을 수 있다
```

#### `in`의 제약

`in` 타입 파라미터는 **파라미터 위치에서만** 사용 가능하다.

```kotlin
interface Consumer<in T> {
    fun consume(value: T)   // OK
    fun produce(): T        // 컴파일 에러
}
```

`Comparator`가 대표적인 반공변 예다.

```kotlin
interface Comparator<in T> {
    fun compare(a: T, b: T): Int
}

val anyComparator: Comparator<Any> = ...
val stringComparator: Comparator<String> = anyComparator   // OK
```

`Any`를 비교할 수 있는 비교자는 당연히 `String`도 비교할 수 있다.

### 변성 요약: out, in, invariant

| 선언 | 변성 | A:B 관계 시 | 사용 위치 |
|---|---|---|---|
| `Foo<T>` | 무공변(invariant) | 관계 없음 | 어디든 |
| `Foo<out T>` | 공변(covariant) | `Foo<A> : Foo<B>` | out 위치만 |
| `Foo<in T>` | 반공변(contravariant) | `Foo<B> : Foo<A>` | in 위치만 |

기억법: **out은 생산자(producer), in은 소비자(consumer).** PECS(Producer-Extends, Consumer-Super) 원칙의 코틀린 버전.

### 사용 지점 변성

지금까지 본 `out`/`in`은 **선언 지점 변성(declaration-site variance)** 이다. 클래스 선언에서 한 번 정하면 모든 사용처에서 그 변성이 적용된다.

자바에는 이 기능이 없다. 자바는 사용 시점마다 와일드카드(`? extends T`, `? super T`)를 적어야 한다. 이를 **사용 지점 변성(use-site variance)** 이라 한다.

코틀린도 사용 지점 변성을 지원한다. 이미 무공변으로 선언된 타입을 특정 사용처에서만 공변/반공변으로 쓰고 싶을 때 유용하다.

```kotlin
fun <T> copyData(source: MutableList<out T>, destination: MutableList<T>) {
    for (item in source) {
        destination.add(item)
    }
}
```

`MutableList`는 원래 무공변이지만, `source` 자리에서는 `out T`로 선언해서 공변처럼 사용한다. 이러면 `MutableList<String>`을 `source`로 넘길 수 있다 (`T`가 `Any`라도).

자바 와일드카드와의 대응 관계:

| 자바 | 코틀린 (사용 지점) |
|---|---|
| `List<? extends T>` | `List<out T>` |
| `List<? super T>` | `List<in T>` |

### 스타 프로젝션

타입 인자에 대한 정보가 없거나 신경 쓰지 않을 때 `*`을 사용한다.

```kotlin
fun printSize(list: List<*>) {
    println(list.size)        // OK (T와 무관)
    val first = list.first()  // first의 타입은 Any?
}
```

`List<*>`은 `List<Any?>`와 비슷하지만 다르다.

- `List<Any?>`: 의도적으로 모든 타입을 담을 수 있는 리스트.
- `List<*>`: **어떤 타입인지 모르지만**, 일관된 한 가지 타입의 리스트.

`List<*>`에서 요소를 읽으면 `Any?`로 받지만, 거기에 새 요소를 쓸 수는 없다(소비 위치). 만약 그것이 `MutableList<*>`이라도 마찬가지다 — 실제 타입이 `MutableList<String>`일 수 있는데 `Int`를 넣으면 깨지니까.

```kotlin
val mutableList: MutableList<*> = mutableListOf("a", "b")
val first = mutableList.first()   // Any?
mutableList.add("c")              // 컴파일 에러
```

스타 프로젝션은 **타입을 모르고도 안전하게 다루기 위한 장치**다.

### 타입 별명 (typealias)

긴 제네릭 타입이나 함수 타입에 짧은 별명을 붙일 수 있다. `typealias` 키워드 사용.

```kotlin
typealias NameMap = Map<String, Person>
typealias ClickHandler = (View, Int) -> Unit

fun register(handler: ClickHandler) { ... }
```

타입 별명은 **컴파일 시점에 원래 타입으로 치환**됨

```kotlin
val m1: NameMap = ...
val m2: Map<String, Person> = ...
// m1과 m2의 타입은 동일
```

#### 활용 예

- **함수 타입에 의미 있는 이름**: `(Int, Int) -> Boolean` → `typealias Comparator2 = (Int, Int) -> Boolean`
- **복잡한 제네릭 단축**: `Map<String, List<Pair<Int, String>>>` → `typealias UserIndex = ...`
- **플랫폼별 구현**: `expect typealias`로 멀티플랫폼에서 활용

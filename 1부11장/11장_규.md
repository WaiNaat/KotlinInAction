# 정리

- 제네릭스
  - 타입 파라미터를 받는 타입을 정의
  - 인스턴스가 만들어질 때 타입 파라미터를 구체적인 타입 인자로 치환
- 자바
  - 제네릭이 늦게 도입됨
  - 하위 호환성을 위해 타입 인자가 없는 제네릭 타입 사용 가능
- 코틀린
  - 제네릭 타입의 타입 인자를 프로그래머가 명시하거나 컴파일러가 추론할 수 있어야 함
  - 제네릭 타입의 타입 인자를 항상 정의해야 함
- 제네릭 함수
  - 자기 자신이 타입 파라미터를 받음
  - 호출할 때 반드시 구체적인 타입 인자를 넘겨야 함
  - 실제로는 대부분 컴파일러가 타입 인자를 추론할 수 있음
- 제네릭 확장 프로퍼티
  - 확장 프로퍼티에만 제네릭 사용 가능
  - 일반 클래스 프로퍼티에 여러 타입의 값을 지정할 수는 없기 때문
- 제네릭 클래스/인터페이스
  - 제네릭 클래스를 확장하는 클래스 또는 제네릭 인터페이스를 구현하는 클래스를 정의하려면 기반 타입의 제네릭 파라미터에 대한 타입 인자를 지정해야 함
  - 구체적인 타입을 넘기거나 타입 파라미터로 받은 타입을 다시 넘길 수 있음
  - 하위 클래스에서 상위 클래스에 정의된 함수를 오버라이드하거나 사용하려면 타입 인자를 구체적인 타입으로 치환해야 함
  - 하위 클래스가 본인의 타입 파라미터를 정의하면서 그것을 기반 클래스의 타입 인자로 사용할 수 있음
  - 클래스가 자기 자신을 타입 인자로 참조할 수도 있음 (예. Comparable)
- 타입 파라미터 제약
  - 클래스나 함수에 사용할 수 있는 타입 인자를 제한하는 기능
  - 어떤 타입을 제네릭 타입 파라미터의 upper bound로 지정하면 그 제네릭 타입 파라미터를 인스턴스화할 때 사용하는 타입 인자는 반드시 그 upper bound 또는 그 하위 타입이어야 함
  - upper bound가 없을 때는 `Any?`로 침
- 자바와의 상호운용
  - `T`는 `Any?`인데 파라미터에 `@NotNull` 어노테이션이 있는 자바 인터페이스를 구현할 때
  - 코틀린에서는 제네릭 파라미터를 정의한 위치가 아니라 타입을 사용하는 지점에서 `T & Any`로 제약을 걸 수 있음
- 타입 소거
  - JVM의 제네릭스는 보통 타입 소거를 사용하도록 구현함
  - 코드의 실행 시점에 제네릭 클래스의 인스턴스에 타입 관련 정보가 없다는 뜻
  - 한계
    - 실행 시점에 타입 인자를 검사할 수 없음
    - 파라미터의 타입 인자에 따라 서로 다른 로직을 구현하고 싶을 때 문제가 됨
  - 장점
    - 저장해야 하는 타입 정보의 크기가 줄어들어 메모리 사용량을 아낄 수 있음
  - `as` 나 `as?` 구문에도 제네릭 타입을 사용할 수는 있으나, 실행 시점에는 무조건 성공하므로 주의해야 함
- 타입 파라미터 실체화
  - 인라인 함수의 타입 파라미터는 실체화됨
  - 실행 시점에 인라인 함수의 실제 타입 인자를 알 수 있음
  - 인라인 함수의 타입 파라미터를 `reified`로 지정하면 실행 시점에 타입이 타입 파라미터로 받은 값의 인스턴스인지를 검사 가능
  - 인라인 함수의 크기가 커지면 문제가 될 수 있으므로 항상 예의주시하기 (noinline 사용 고려)
- 인라인 함수에서만 실체화가 가능한 이유
  - 컴파일러는 인라인 함수의 본문을 구현한 바이트코드를 해당 함수가 호출되는 모든 지점에 삽입
  - 컴파일러가 타입 인자로 쓰인 구체적인 클래스를 참조하는 바이트코드 역시 생성해서 삽입할 수 있음
  - 바이트코드는 파라미터가 아니라 실제 타입이므로 타입 소거의 영향을 받지 않음
- 타입 파라미터 실체화의 예시
  - 표준 라이브러리 함수 `filterIsInstance`에서 사용중
  - `java.lang.Class` 타입 인자를 파라미터로 받는 API에 대한 유틸성 로직을 만드는 경우
- 실체화된 타입 파라미터의 제약
  - 가능
    - 타입 검사와 캐스팅
    - 코틀린 리플렉션 API `::class`
    - 코틀린 타입에 대응하는 java class를 얻어야 하는 경우
    - 타른 함수를 호출할 때 타입 인자로 사용
  - 불가능
    - 타입 파라미터 클래스의 인스턴스 생성하기
    - 타입 파라미터 클래스의 companion object 호출하기
    - 실체화된 타입 파라미터를 요구하는 함수를 호출하면서, 실체화하지 않은 타입 파라미터로 받은 값을 타입 인자로 넘기기
    - 클래스, 프로퍼티, 인라인 함수가 아닌 함수의 타입 파라미터를 reified로 지정하기
- 변성(variance)
  - 기저 타입이 같고 타입 인자가 다른 여러 타입이 서로 어떤 관계가 있는지 설명하는 개념
  - 코드에서 위험할 여지가 있는 메서드를 호출할 수 없게 만들어서 제네릭 타입의 인스턴스 역할을 하는 클래스 인스턴스를 잘못 사용하는 것을 방지하는 역할
- 타입과 클래스
  - 제네릭 클래스가 아닌 경우
    - 클래스명이 곧 타입
    - 클래스명을 nullable type에도 쓸 수 있음
    - 모든 코틀린 클래스는 적어도 둘 이상의 타입을 구성할 수 있음
  - 제네릭 클래스
    - 올바른 타입을 얻으려면 제네릭 타입의 타입 파라미터를 구체적인 타입 인자로 바꿔야 함
    - 각각의 제네릭 클래스는 무수히 많은 타입을 만들어낼 수 있음
- 하위 타입
  - 타입 A의 값이 필요한 모든 장소에 어떤 타입 B를 넣어도 아무 문제가 없다면 B는 A의 하위 타입
  - 모든 타입은 자기 자신의 하위 타입
  - 상위 타입: 하위 타입의 반대
  - 컴파일러는 변수 대입, 함수 인자 전달 시 항상 하위 타입 검사 수행
  - 어떤 값의 타입이 변수 타입의 하위 타입인 경우에만 값을 변수에 대입 가능
  - 어떤 인터페이스를 구현하는 클래스의 타입은 그 인터페이스 타입의 하위 타입
  - non-nullable 은 nullable의 하위 타입. (하지만 클래스는 같음)
- 무공변(invariance)
  - 임의의 제네릭 클래스 X에 대해 `X<A>`와 `X<B>`가 서로의 상위 타입도 아니고 하위 타입도 아닌 경우 X는 타입 파라미터에 대해 무공변
- 공변(covariance)
  - A가 B의 하위 타입일 때 `X<A>`가 `X<B>`의 하위 타입인 경우
  - 하위 타입 관계를 유지한다는 뜻
  - 제네릭 클래스가 타입 파라미터에 대해 공변적임을 표시하려면 타입 파라미터 이름 앞에 `out`을 붙여야 함
  - 모든 클래스를 공변적으로 만들 수는 없음
  - 타입 안정성을 보장하기 위해 공변적 파라미터는 항상 out 위치에만 있어야 함
  - 클래스가 해당 타입 파라미터 타입의 값을 생산할 수는 있지만 소비할 수는 없음
- in, out
  - 함수의 반환 타입에 쓰이면 out
  - 함수의 파라미터 타입에 쓰이면 in
  - 생성자 파라미터는 in, out 둘 다 아님
    - 생성자는 인스턴스가 생성된 후에 호출할 수 없기 때문
    - 단, 생성자 파라미터에 val, var를 쓴다면 getter, setter를 만드는 것이므로 in과 out의 영향을 받음
  - 비공개 메서드의 파라미터는 in, out 둘 다 아님
    - 변성은 외부 사용자가 클래스를 잘못 쓰지 않게끔 하기 위한 것이므로
- 반공변(contravariance)
  - A가 B의 하위 타입일 때 `X<B>`가 `X<A>`의 하위 타입인 경우
  - 하위 타입 관계가 반대임
  - 타입 파라미터를 in 위치에서만 사용할 수 있음
- 선언 지점 변성
  - 클래스를 선언하면서 변성을 지정
- 사용 지점 변성
  - 타입 파라미터가 있는 타입을 사용할 때마다 변성을 명시하는 것
- 타입 프로젝션
  - 타입 파라미터를 사용하는 위치라면 어디에나 in 또는 out을 붙여서 사용에 제약을 가할 수 있음
- 스타 프로젝션
  - `MutableList<*>`는 어떤 정해진 구체적인 타입의 원소만을 답는 리스트지만, 그 원소의 정확한 타입은 모른다는 뜻
  - 타입 인자에 대한 정보가 중요하지 않을 때 스타 프로젝션을 사용
  - 타입 파라미터를 시그니처에서 전혀 언급하지 않거나, 데이터를 읽기는 하지만 구체적인 타입은 신경쓰지 않을 때 (예. 배열 로깅)
- type alias
  - 타입에 대해 다른 키워드를 부여
  - 함수형 타입에 맥락 부여 가능
  - 처음 보는 사람은 정의를 이해하기 위한 시간이 필요
  - 컴파일러 입장에서는 타입 안정성에 아무런 기여를 하지 않음
    - 최소한의 비용으로 타입 안정성을 보장하기 위해서는 인라인 클래스 추천


# 궁금했던 것들

### 왜 타입 파라미터가 두 개일 때는 `Map<K, V>` 처럼 K와 V를 많이 쓸까?
- Key, Value

### 타입 파라미터에 여러 제약을 걸 때 (`fun <T> test(target: T) where T: Comparable, T: Appendable {}`) 조건이 AND 로 걸리는 듯하다. OR로 거는 방법은 없을까?
- 없음
- 실행 시점에 A에는 있고 B에는 없는 메서드 등을 호출하기 난감하기 때문
- 대안: sealed class, function overloading, Any+when

### `@NotNull`에 대처하기 위한 `T & Any`가 있다. `K & V`, `T & String` 같은 형태도 가능한지?
- 불가능
- 자바 호환성만을 위한 `T & Any` 라는 특수기가 있는 것 뿐임.

### 어떤 위치가 out이고 in인지 판정하는 정확한 알고리즘이 궁금하면 코틀린 언어 문서를 참고하라는데, 그렇게 복잡한가?
- val, var, counstructor 처럼 외워야 하는 규칙이 있음
- 고차함수 들어가면 쉽지 않음
  - `fun play(action: (T) -> Unit)`
  - play 내부에서 action() 을 한다고 쳤을 때, T는 action()의 인자로 넘어감. play 입장에서는 action 이라는 외부에 전달해줘야 하는 값이 T이므로 이 T는 out

### Q1) 인라인 함수에서만 실체화가 가능한 이유가 무엇일까요? (p.489)
- 컴파일러는 인라인 함수의 본문을 구현한 바이트코드를 해당 함수가 호출되는 모든 지점에 삽입
- 컴파일러가 타입 인자로 쓰인 구체적인 클래스를 참조하는 바이트코드 역시 생성해서 삽입할 수 있음
- 바이트코드는 파라미터가 아니라 실제 타입이므로 타입 소거의 영향을 받지 않음

### Q2) `MyClass<*>`랑 `MyClass<Any?>`의 차이는 무엇일까요? (p.513)
- `*`
  - 어떤 구체적인 타입이기는 한데, 그게 정확히 뭔지는 모르고, 몰라도 된다는 것을 의미
- `Any?`
  - 실제로 구체적인 타입이 `Any?` 라는 이야기.

### Q1) 코틀린에서 일반 프로퍼티는 제네릭으로 만들 수 없는데, 확장 프로퍼티는 왜 가능한가?
- 일반 클래스 프로퍼티는 이미 하나의 타입을 갖도록 정해져 있는 상태
- 일반 클래스 프로퍼티에 여러 타입의 값을 지정할 수는 없기 때문
- 제네릭 클래스의 경우에는 확장 프로퍼티를 만들 때 제네릭 파라미터를 받아야 실제 어떤 타입의 값을 갖는지 지정할 수 있음
    
### Q2) typealias와 inline value class는 어떻게 다른가?
- type alias
  - 타입에 이름을 부여
  - 일종의 이름 바꿔치기에 가까운 개념이라 컴파일러에게 어떠한 제약도 주지 않음
- value class
  - 하나의 값에 대해서 런타임에는 값으로 동작하지만, 컴파일 타임에는 별도의 클래스로 동작할 수 있게끔 해 주는 장치
  - 성능 오버헤드 없이 값에 의미를 부여하면서, 발생 가능한 오류들을 컴파일 타임에 잡고 싶을 때 사용

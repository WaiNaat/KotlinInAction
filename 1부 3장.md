### Q1) `vararg` 의 비용은 어떻게 될까? 
    - p154, 어떻게 동작하는가에 대해 파악하기 위한 질문

**A) (김규)**

- `vararg` 는 배열로 취급함
- `vararg` 를 호출할 때마다 내부적으로는 새로운 배열 객체를 생성
    - 반복문 등에서 큰 배열을 스프레드 연산자를 이용해서 넘기는 행위 등을 하면 불필요한 배열 생성으로 인해 비효율적일 수 있음
- 컴퓨팅 측면에서는 이득이 없는 구조로 보이는데, 사용하는 입장에서는 간단한 것에도 listOf 를 하지 않아도 되기 때문에 가독성이나 생산성 측면에서 유리한 점이 있을 것 같음.

### Q2) 확장 함수의 장점과 단점은 무엇일까?
(책에는 없는 내용: 자바스크립트에서는 이걸 쓰지 않는게 보편적인 컨벤션인데 코틀린에서는 권장하길래 궁금해서 질문남겨요)

**A) (김세진)**

- 장점
    - 확장은 기존 클래스를 수정하지 않고 같은 문법으로 호출 가능
    - 외부 라이브러리/표준 라이브러리 등에 의미 있는 API를 붙일 수 있다.
    - 네임스페이스 정리에 좋다
- 단점
    - 진짜 멤버가 아니라서 헷갈린다.
    - 과도하면 API 탐색성이 떨어진다
        - 어디서 정의됐는 지 찾기 어렵고, import에 따라 보이는 함수가 다르고, 같은 이름이 여러개여서 이름 충돌 위험 등
- JavaScript에서 흔히 경계하는 건 **prototype 오염**이다. 즉 전역적으로 내장 객체의 동작을 바꿔 버릴 수 있기 때문에 위험하다.
반면 Kotlin의 확장 함수는 **클래스를 실제로 수정하지 않는다**. 그냥 호출 문법만 멤버처럼 보이게 할 뿐이다. 그래서 JS의 monkey patching과는 본질적으로 다르다. Kotlin 문서도 “확장은 클래스를 수정하지 않는다”고 명시한다.
즉 Kotlin 확장은 “언어 차원의 안전한 문법 설탕”에 가깝고, JS prototype extension은 런타임 객체 모델 자체를 건드릴 수 있다는 차이가 있다.

### Q1) 로컬 함수로 분리하면 중복을 없애며 깔끔하게 유지할 수 있다. private 메소드 선언대비 로컬 함수가 더 깔끔한 것이 맞을까?, 맞다면 비교하는 예제 코드를 작성해주세요.

**A) (김세진)**

private 선언

```kotlin
class UserService {

    fun saveUser(user: User) {
        validateName(user)
        validateAge(user)
        // 저장 로직
    }

    private fun validateName(user: User) {
        require(user.name.isNotBlank()) { "이름은 비어 있을 수 없습니다." }
    }

    private fun validateAge(user: User) {
        require(user.age >= 0) { "나이는 0 이상이어야 합니다." }
    }
}
```

로컬 함수

```kotlin
class UserService {

    fun saveUser(user: User) {
        fun validate(value: String, fieldName: String) {
            require(value.isNotBlank()) { "$fieldName 은(는) 비어 있을 수 없습니다." }
        }

        fun validateAge(age: Int) {
            require(age >= 0) { "나이는 0 이상이어야 합니다." }
        }

        validate(user.name, "이름")
        validateAge(user.age)

        // 저장 로직
    }
}
```

- 토비의 스프링 AI 강의 추천
- 로컬 함수 사용 시
    - 오직 한 함수 안에서만 쓰이고,
    - 바깥 함수의 지역 변수 등을 그대로 활용하고,
    - 변수를 가볍게 점검하는 용도
- private 메소드 사용 시
    - 함수가 너무 길어서 메소드가 복잡해지거나,
    - 여러곳에서 사용하여 재사용성이 있어 메소드로 쓰는 게 낫거나,
    - 독립 개념으로 이름 붙일 가치가 있으면 private

### Q2) `listOf()`, `setOf()`, `mapOf()`로 생성한 컬렉션의 실제 런타임 타입은 무엇인가? 코틀린 컬렉션이 자바 컬렉션과 같은 클래스를 쓴다면, `list.add()`는 왜 컴파일 에러가 나는가?

**A) (김규)**

```kotlin
println(listOf<Int>().javaClass.name)      // kotlin.collections.EmptyList
println(listOf(1).javaClass.name)          // java.util.Collections$SingletonList
println(listOf(1, 2).javaClass.name)       // java.util.Arrays$ArrayList

println(setOf<Int>().javaClass.name)       // kotlin.collections.EmptySet
println(setOf(1).javaClass.name)           // java.util.Collections$SingletonSet
println(setOf(1, 2).javaClass.name)        // java.util.LinkedHashSet

println(mapOf<Int, Int>().javaClass.name)  // kotlin.collections.EmptyMap
println(mapOf(1 to 1).javaClass.name)      // java.util.Collections$SingletonMap
println(mapOf(1 to 1, 2 to 2).javaClass.name) // java.util.LinkedHashMap
```

- empty 는 각자 본인만의 empty 전용 객체가 있어서 그것을 사용
- list
    - java.util.Arrays$ArrayList
    - 정해진 크기의 배열이라 변경 불가능
- mutableList
    - java.util.ArrayList
    - 가변 크기
- set, mutableSet
    - java.util.LinkedHashSet
    - 크기 1의 set 은 java.util.Collections$SingletonSet 라는게 따로 있음
- map, mutableMap
    - java.util.LinkedHashMap
    - 크기 1의 map 은 java.util.Collections$SingletonMap 라는게 따로 있음
- 인터페이스를 분리해서 list, mutableList 를 따로 관리하기 때문에 컴파일 에러 발생 (immutability를 중시하는 컨셉)

### Q3) Kotlin에서 `+` 연산이나 `${}` 템플릿을 써도 성능상 괜찮은가? `StringBuilder`를 명시적으로 써야 하는 경우는?

**A) (김세진)**

- 단발성 문자열 결합에서 `+`나 `${}`는 보통 성능상 문제 없다. 하지만 **루프 안 누적 결합**이나 **큰 문자열의 반복 생성**에서는 `StringBuilder` 또는 `buildString`을 쓰는 게 맞다. 아래 코드는 매 반복마다 새로운 문자열을 만든다. `String`은 immutable이기 때문에 누적 결합은 비용이 커진다.

```kotlin
var result = ""
for (i in 1..10000) {
result += i
}
```

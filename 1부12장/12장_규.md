# 정리
- 어노테이션과 리플렉션
  - 미리 알지 못하는 임의의 클래스를 다룰 수 있음
  - 어노테이션: 라이브러리가 요구하는 의미를 클래스에 부여
  - 리플렉션: 실행 시점에 컴파일러 내부 구조를 분석
- 어노테이션
  - `@`과 어노테이션 이름을 선언 앞에 넣으면 됨
  - 어노테이션의 인자
    - 기본 타입의 값, 문자열, 이넘, 클래스 참조, 다른 어노테이션 클래스 가능
    - 위에 적힌 값들의 배열도 가능
    - 어노테이션의 인자는 컴파일 시점에 알 수 있어야 함
    - 임의의 프로퍼티를 인자로 지정 불가. 반드시 `const`만 가능
- use-site target
  - 코틀린 프로퍼티는 자바의 게터, 세터, 필드, 파라미터 선언, 경우에 따라 생성자 파라미터와 대응함
  - 얘네들 중 정확히 어디에 어노테이션을 붙일지 표시할 필요가 있음
  - 자바에 선언된 어노테이션을 프로퍼티에 붙이면 기본적으로 프로퍼티의 필드에 붙음
  - 코틀린으로 어노테이션 선언 시 프로퍼티 자체에도 지정할 수 있음
  - property, field, gfet, set, receiver, param, setparam, delegate, file 값이 가능함
- 어노테이션의 예시
  - 객체 직렬화 제어
    - 직렬화: 객체를 저장 장치에 저장하거나 네트워크를 통해 전송하기 위해 텍스트나 이진 형식으로 변환하는 것
    - 역직렬화: 텍스트나 이진 형식으로 저장된 데이터에서 원래의 객체를 만들어내는 것
- 어노테이션 선언하기
  - `annotation class`
  - 어노테이션 클래스는 내부에 아무런 코드도 들어있을 수 없음\
  - 컴파일러 자체가 본문을 정의하지 못하게 막음
  - 파라미터가 있는 어노테이션 정의 시 주 생성자에 반드시 모든 파라미터를 `val`로 선언해야 함\
- 메타 어노테이션
  - 어노테이션 클래스에 적용할 수 있는 어노테이션
  - 컴파일러가 어노테이션을 처리하는 방법 제어
  - `@Target`
    - 어노테이션을 적용할 수 있는 요소의 유형(프로퍼티, 클래스, 메서드, 필드 등등)을 정함
  - `@Retention`
    - 어노테이션 코드를 어디까지 유지할지 정의
    - 소스, `.class`, 실행 시점 등
    - 자바: `.class`에는 저장하는데 실행시점에는 못씀
    - 코틀린: 기본적으로 런타임에도 사용가능함 
- 클래스를 어노테이션 파라미터로 사용
  - 클래스 이름 뒤에 `::class`를 붙여서 클래스를 가리킴
  - `KClass`는 코틀린 클래스에 대한 참조를 저장함
  - `KClass<out 클래스명<*>>`
- 리플렉션
  - 실행 시점에 동적으로 객체의 프로퍼티와 메서드에 접근할 수 있게 해주는 방법
  - 객체가 제공하는 메서드나 프로퍼티 이름을 오직 실행 시점에만 알 수 있는 경우에 사용
  - 코틀린 리플렉션 API 또는 자바 리플렉션 API 사용: 코틀린 리플렉션 API는 별도 라이브러리로 제공중
- `KClass`
  - 클래스 안의 모든 선언
  - 각 선언에 접근
  - 클래스의 상위 클래스 확인
  - `MyClass::class`는 `KClass`의 인스턴스를 반환함
  - 익명 클래스는 `simpleName`, `qualifiedName`이 null이므로 조심
- `KCallable`
  - 함수(`KFunction`)와 프로퍼티(`KProperty`)를 아우르는 공통 상위 인터페이스
  - `call`
    - getter 호출, 인자는 vararg로 전달
    - default parameter 를 제공하지 않음
  - `callBy`
    - default parameter가 있다면 적용할 수 있음
    - 파라미터 순서도 지킬 필요 없음
- `KFunctionN`
  - N은 파라미터의 개수
  - `invoke` 메서드로 `call`과는 다르게 타입 안전한 메서드 사용 가능
  - 컴파일러가 생성한 합성 타입이라 리플렉션 패키지에 없음
- `KProperty`
  - `KProperty0`, `KMutableProperty0`
    - 최상위 프로퍼티는 이 인터페이스의 인스턴스로 표현됨
    - 인자가 없는 `get` 메서드 제공
  - `KProperty1`, `KMutableProperty1`
    - 멤버 프로퍼티는 이 인터페이스의 인스턴스로 표현됨
    - 인자가 1개인 `get` 메서드 제공(인자는 값을 얻을 수신 객체 인스턴스)
  - 최상위 수준이나 클래스 안의 프로퍼티만 리플렉션으로 접근 가능
  - 함수의 로컬 변수에는 접근 불가능
- `KAnnotatedElement`
  - 해당 요소에 적용된 모든 어노테이션 인스턴스 보유
  - `findAnnotation`으로 원하는 어노테이션을 얻을 수 있음

# 궁금했던 것들

### `val`도 변경 불가능한 값인데 왜 어노테이션에 쓰려면 `const val`이어야만 하는가?
- `val`은 runtime read-only
- `const`는 compile-time constant. 컴파일러가 값을 인라이닝함

### 예전에 AOP에 어노테이션이 좋다고 들었는데 진짜 좋은게 맞는가? 상속의 개념이 아니라서 좋다고 하는 것인가?
- 상속 구조에 얽매이지 않고 비즈니스 로직에만 집중할 수 있게 해줌

### 540쪽, `KClass<out Any>`를 써야만 하는 이유?
- `KClass<Any>`는 하위 타입 관계를 유지하지 않으므로 내가 넣고싶은 `MyClass`가 `Any`의 하위라는 것을 보장할 수 없음. 따라서 `Any::class`만 가능함
- `KClass<out Any>`는 하위 타입 관계를 유지함(공변성) -> `MyClass`가 `Any`이기도 하다는 것을 알고 있어서 쓸 수 있음

### `KFunctionN`에서 N의 최댓값은?
- 22
- 코틀린 컴파일러가 내부적으로 22까지 미리 만들어서 들고있다가 필요한걸 넣어줌
- 23개 이상부터는 새로운 인터페이스를 만들거나, 인자를 배열로 만들어서 넣는 등 다른 조치를 취하기 때문에 코드 작성이 가능은 함

### `KProperty2`는 없는가?
- 있음
- 어떤 클래스 내부에 작성된 다른 클래스의 확장 프로퍼티
- 본인과 확장 대상 인스턴스가 필요해서 두 개의 인자를 받음
- `KProperty3`은 없음

### 클래스의 인스턴스라면 `KClass`는 '무조건' 존재하는가?
- ㅇㅇ
- 다만 `value class`처럼 런타임에 없어지는 애들은 원하는 대로 동작하지 않을 수는 있음

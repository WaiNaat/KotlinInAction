### Q1) smart cast는 항상 발생하지 않는데, 발생하는 조건과 이유는?

**A) 김규**

p.110

* `val` 변수이거나 클래스 프로퍼티처럼 **값이 변경되지 않는 것이 보장될 때** smart cast 발생
* `is`로 타입 검사 후 컴파일러가 타입이 유지된다고 판단하면 자동 캐스팅

**smart cast가 안되는 경우**

* `var` (값 변경 가능)
* 커스텀 getter
* 함수 반환값

→ 이유: 컴파일러가 **값이 중간에 바뀔 가능성**을 배제할 수 없기 때문

### Q2) `enum.ordinal`을 비즈니스 로직이나 DB 값으로 사용하면 위험한 이유는?

**A) 김승수**

* enum은 `name`과 `ordinal` 프로퍼티를 가짐

  * name → 이름
  * ordinal → 선언 순서

문제:
→ enum 순서 변경 시 ordinal 값이 바뀜

```kotlin
enum class OrderStatus {
    PENDING,    // ordinal = 0
    PAID,       // ordinal = 1
    SHIPPED,    // ordinal = 2
    DELIVERED   // ordinal = 3
}
```

요구사항 변경:

```kotlin
enum class OrderStatus {
    PENDING,    // ordinal = 0
    CANCELLED,  // ordinal = 1
    PAID,       // ordinal = 2 (기존 1)
    SHIPPED,    // ordinal = 3
    DELIVERED   // ordinal = 4
}
```

→ DB 저장값과 매칭 깨짐
→ 기존 데이터 전체 오류 발생 가능

**결론**

* ordinal 사용 금지
* name 또는 별도 code 필드 사용 권장

### Q3) Kotlin에서 `if`, `when`, `try`가 모두 식(expression)이라는 점의 장점은?

**A) 김승수**

### 1. if → 삼항 연산자 불필요

### 2. when → 값을 반환하는 패턴 매칭

❌ Statement 스타일

```kotlin
var message: String

when (status) {
    OrderStatus.PAID -> message = "결제 완료"
    OrderStatus.SHIPPED -> message = "배송 중"
    else -> message = "알 수 없음"
}
```

✅ Expression 스타일

```kotlin
val message = when (status) {
    OrderStatus.PAID -> "결제 완료"
    OrderStatus.SHIPPED -> "배송 중"
    else -> "알 수 없음"
}
```

장점:

* val 사용 가능
* 재할당 제거
* 가독성 증가

추가 장점:

* else 강제 → 모든 케이스 처리 보장
* sealed class 사용 시 else 없이 완전성 체크 가능

```kotlin
val message = when (result) {
    is Result.Success -> result.data
    is Result.Error -> result.message
}
```

### 3. try → 결과값 바로 할당 가능

Java:

```java
int parsed;

try {
    parsed = Integer.parseInt(input);
} catch (NumberFormatException e) {
    parsed = 0;
}
```

Kotlin:

```kotlin
val parsed = try {
    input.toInt()
} catch (e: NumberFormatException) {
    0
}
```

→ 변수 사전 선언 불필요
→ 코드 간결

### Q1) 코틀린에서는 대입이 항상 문(statement)이다. Kotlin vs Java 대입이 식인 예제는?

**A) 김세진**

Kotlin:

```kotlin
val x = (y = 5) // 에러
println(y = 5) // 에러
```

Java:

```java
int a, b;

a = b = 10; // 가능
```

차이:

* Java → 대입이 expression
* Kotlin → 대입이 statement

이유:
→ 실수로 조건문에서 대입하는 버그 방지

(Java 위험한 코드)

```java
if (a = b) // 실수 가능
```

Kotlin은 구조적으로 불가능하게 설계됨.

### Q2) `isSquare`를 함수가 아닌 프로퍼티로 만드는 코틀린 컨벤션 기준은?

**A) 김세진**

기준:

**프로퍼티로 만드는 경우**

* 객체 상태 표현
* 계산 비용 작음
* 부작용 없음
* 클래스 특성 표현

```kotlin
val isSquare: Boolean
    get() = height == width
```

**함수로 만드는 경우**

* 계산 비용 큼
* 로직 복잡
* 파라미터 필요
* 행동 표현

정리:

| 구분   | 프로퍼티 | 함수      |
| ---- | ---- | ------- |
| 의미   | 상태   | 행동      |
| 계산비용 | 작음   | 클 수 있음  |
| 부작용  | 없음   | 있을 수 있음 |
| 파라미터 | 없음   | 있음 가능   |

### Q3) 아래 `eval` 함수에서 스마트 캐스트가 동작하는 이유는?

**A) 김세진**

```kotlin
fun eval(e: Exp): Int {
    if (e is Num) return e.num
    if (e is Sum) return eval(e.right) + eval(e.left)

    throw IllegalArgumentException("지원하지 않는 타입")
}
```

이유:

* `e is Num`
* `e is Sum`

에서 타입 검사가 발생했기 때문에

→ 컴파일러가 이후 타입을 확정 가능

결과:

→ 명시적 캐스팅 없이 내부 프로퍼티 접근 가능

### Q1) 코틀린에서 checked / unchecked exception을 구분하지 않는 이유는?

**A) 김세진**

p.124

이유:

* 실제로 처리 불가능한 exception도 강제 처리 필요

Java:
→ 컴파일러가 exception 처리 강제

Kotlin:
→ 개발자 책임
→ 예상 가능한 에러는 타입으로 모델링

예:

* sealed class
* Result 타입

정리:

| Java                 | Kotlin |
| -------------------- | ------ |
| checked exception 강제 | 없음     |
| try-catch 많음         | 필요한 곳만 |
| 컴파일러 중심              | 개발자 중심 |

### Q3-1) 반복문 내에서 이터레이션이 가능한 조건은?

**A) 김세진**

핵심:

→ `iterator()` 제공해야 함

Kotlin for 문 동작:

```kotlin
for (x in list) {
    println(x)
}
```

컴파일 후:

```kotlin
val iterator = list.iterator()

while (iterator.hasNext()) {
    val x = iterator.next()
    println(x)
}
```

즉:

→ for는 iterator 기반 문법적 설탕

Iterator 인터페이스:

```kotlin
interface Iterator<out T> {
    operator fun next(): T
    operator fun hasNext(): Boolean
}
```

정리:

반복 가능 조건:

* iterator()
* hasNext()
* next()

기본 제공:

* List
* Set
* Map
* IntRange

직접 구현 가능:

예: LocalDate Range

```kotlin
operator fun ClosedRange<LocalDate>.iterator(): Iterator<LocalDate> =
    object : Iterator<LocalDate> {

        var current = start

        override fun hasNext() =
            current <= endInclusive

        override fun next(): LocalDate {
            val result = current
            current = current.plusDays(1)
            return result
        }
    }
```

→ iterator 구현하면 for 사용 가능

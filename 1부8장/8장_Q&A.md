### Q1) 책에서 음이 아닌 정수가 필요할 때 부호 없는 정수를 사용하는 것은 코틀린의 부호 없는 정수의 목표가 아니라고 하는데, 그럼 목표가 도대체 뭘까요... (p.352 관련)

**A)김세진**

- 답변... 소스코드 찾아보고 있으면 공유
- **목표:** 음수가 없는 도메인 표현이 아니라, **비트 단위 연산**이나 **네이티브 라이브러리(C/C++)와의 데이터 매핑**이 주 목적입니다.
- **이유:** 만약 `UInt`를 "나이" 변수에 썼는데, 계산 실수로 `1`이 되는 순간 이 값은 에러가 나는 게 아니라 `4,294,967,295`라는 거대한 양수로 변해버립니다. 이는 논리적 오류를 찾기 더 어렵게 만듭니다. 음수를 막고 싶다면 커스텀 클래스(Value Class 등)나 `require()` 체크를 사용하는 것이 코틀린스러운 방식입니다.

### Q1) Kotlin의 List는 자바의 단순 Collection으로 변환된다. 왜 Java의 UnmodifiableCollection을 사용하지 않은 것인가?

**A)김세진**

- `UnmodifiableCollection`으로 감싸는 것은 별도의 래퍼 객체를 생성하는 오버헤드를 발생시킵니다.
- Collection은 메서드를 지원하여 값을 읽을 수 있게 하는데, UnmodifiableCollection는 레퍼를 하나 더 만드는거라,추가 비용이 발생
- 또한 코틀린이 좋아하는 컴파일 시점에 제약을 해야하는데, 이걸 사용하면 런타임 시점에 오류가 날 수 있어 쓰지 않은 것 같음.

### Q2) Nothing을 사용했던 경험이 있으신가요? Nothing은 정말 필요한 타입일까요?

**A)김규**

- 참고: 2장의 궁금했던 것들
  - 코틀린의 throw는 식이다. 변수의 초기화에 throw를 사용할 경우 해당 변수가 초기화되지 않는다. 하지만 '초기화되지 않는다'는 사실을 알아야 하는가?
- 개발자가 직접 쓰라고 만든 타입이 아닙니다
- 컴파일러에게 필요한 타입입니다
  - 예시) throw는 Nothing 타입의 값을 반환
    - 이 때 발생하는 '초기화되지 않는다'는 현상을 알 필요는 없습니다
  - 컴파일러가 필요로 하는 이유
    - Unreachable code 감지
    - smart cast에 사용

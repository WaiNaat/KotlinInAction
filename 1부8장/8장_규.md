# 8장. 기본 타입, 컬렉션, 배열

# 정리

- 자바
  - 원시 타입은 더 효율적으로 저장하고 여기저기 전달할 수 있음
  - 메서드 호출이나 컬렉션에 원시 타입 값을 담을 수 없음
  - `java.lang.Integer` 같은 래퍼 타입으로 원시 값을 감싸서 사용
- 코틀린
  - 원시 타입과 래퍼 타입을 구분하지 않음
  - 그래서 편리함
  - 실행 시점에 가능한 한 가장 효율적인 방식으로 표현
- 부호 없은 숫자 타입
  - JVM의 일반적인 원시 타입을 확장해 부호 없는 타입을 제공
  - 양수를 표현하기 위해 모든 비트 범위를 사용하고 싶을 때 사용
  - 실행 시점에 필요한 때만 래핑됨
  - JVM은 부호가 없는 수에 대한 원시 타입을 제공하지 않음
  - 코틀린은 inline class를 이용해서 제공함 -> 성능상 하자 없음
- null
  - null 참조는 자바에서는 참조 타입의 변수에만 대입할 수 있음
  - 코틀린에서 널이 될 수 있는 원시 타입은 자바의 래퍼 타입으로 컴파일됨
- 제네릭 클래스는 래퍼 타입 사용
- 숫자 타입 변환
  - 코틀린은 overflow, equals 등에서 숫자 타입을 자동 변환해주지 않음: 개발자의 혼란을 피하려는 목적
  - 숫자 리터럴을 타입이 알려진 변수에 대입하거나 함수의 인자로 넘기변 그건 자동으로 변환해줌
- `Any`
  - 모든 널이 될 수 없는 타입(원시 타입 포함)의 조상 타입
  - 널을 포함한 모든 타입을 하려면 `Any?` 사용
  - 내부적으로 `java.lang.Object` 에 대응
- `Unit`
  - 자바 `void` 와 같은 기능
  - Unit이 void와 다른 점
    - 모든 기능을 갖는 일반적인 타입
    - 타입 인자로 사용 가능
    - Unit 타입에 속한 값은 단 하나, Unit임
    - Unit 타입의 함수는 Unit 값을 암시적으로 반환
- `Nothing`
  - 함수가 정상적으로 끝나지 않는다는 것을 표시
  - 아무 값도 포함하지 않음
  - 함수의 반환 타입이나 반환 타입으로 쓰일 타입 파라미터로만 사용 가능
  - Nothing 타입의 변수를 선언하더라도 아무 값도 저장할 수 없음
- `filterNotNull`
  - 널이 될 수 있는 갑시으로 이루어진 컬렉션에서 널 값을 걸러내는 경우에 사랑
- mutability와 collection
  - 코틀린에서는 컬렉션 안의 데이터에 접근하는 인터페이스와 컬렉션 안의 데이터를 변경하는 인터페이스를 분리함
  - 가능하면 읽기 전용 인터페이스를 사용하기
  - 구별한 이유: 프로그램에서 데이터에 어떤 일이 벌어지는지 더 쉽게 이해하기 위함
  - 읽기 전용 컬렉션이더라도 변경 불가능한 컬렉션일 필요가 없음
    - 함수가 바라보는 인터페이스가 immutable 하더라도 내부적으로는 mutable 일 수 있음
    - 멀티스레드 환경에서 고려 필요
  - 자바는 이 둘을 구분하지 않으므로 조심할 것

# 궁금했던 것들

### 음이 아닌 정수가 필요할 때 부호 없는 정수를 사용하는 것은 코틀린의 부호 없는 정수의 목표가 아니라고 하는데, 그럼 왜 이렇게 만든거임?
- 내가 쓰라고 만든게 아님
- low-level 에서 shift, unshift 등을 사용할 때 sign bit 고려하지 않고 사용하기 편함
- C/C++ 환경에서는 unsigned 타입을 많이 써서 호환성을 맞춰주기 위함
- IPv4 값 표현, 시스템 레지스트리 등 sign bit 없는 순수한 값을 다룰 때 사용

### `Unit` 은 `Any` 인가? 아니면 `null` 인가?
- `Unit` 은 singleton object임

### `Nothing` 타입의 변수를 선언하면 메모리를 잡아먹는지?
- 메모리에 값을 할당하는 것 자체가 물리적으로 불가능
- `Nothing?` 이면 null 할당해야 해서 공간이 필요함
- 코드상으로 Nothing 에 할당을 할수있는건 무한루프 또는 throw

### `Nothing` 과 `throw Error`의 차이는?
- throw 식을 실행한 결과물의 타입이 Nothing

### 읽기 전용 컬렉션이 thread-safe 하지 않아서, 다중 스레드 환경에서 데이터를 다루는 경우 그 데이터를 동기화하거나 동시 접근을 허용하는 데이터 구조를 활용해야 한다는데 이런 것들의 예시가 있을지?
- `ConcurrentHashMap`
  - 버킷 단위로 lock을 걸어서 멀티스레드 환경에서 병목 적게 활용 가능
- `CopyOnWriteArrayList`
  - 데이터 변경 시마다 내부 배열의 복사본을 통째로 만듦
  - 읽기는 빠른데 쓰기는 느림
- `PersistentList`
  - 데이터 변경 시마다 새로운 컬렉션이 반환됨
  - 값은 내부적으로 트리 형태로 관리하여 메모리 낭비가 심하지 않음
- `Mutex`
  - mutableList 자체에 lock을 거는 방법
  - lock 걸려있어도 thread 를 block하지 않고 suspend

### List와 Array의 차이?
- Array
  - 고정 크기
  - mutable
  - 성능 확실함
- List
  - 가변 크기
  - mutable과 immutable interface가 구분되어 있음
  - 유틸기 많음

### Q1) Any 와 Generic(T) 차이
- Any
  - 클래스
  - null 이 아닌 것들
  - 타입 캐스팅을 잘못 할 경우 런타임에 오류 발생 가능
- 타입 파라미터 T
  - 변수
  - 별도의 bound 가 없으면 null 을 포함한 모든 타입이 가능
  - 컴파일 타임에 에러를 확인할 수 있어서 타입 안정성 보장에 유리함

### Q2) Array VS List 선택 기준
- Array
  - 고정 크기
  - mutable
  - 성능 확실함 (원시 타입 전용 array가 있어서 boxing이 없는 경우가 있음)
- List
  - 가변 크기
  - mutable과 immutable interface가 구분되어 있음
  - 유틸기 많음

### Q1) Kotlin의 List는 자바의 단순 Collection으로 변환된다. 왜 Java의 UnmodifiableCollection을 사용하지 않은 것인가?
- `UnmodifiableList`: add, remove 메서드가 있으나 호출 시 런타임 에러
- List는 컨셉 자체가 immutable 이 아니라 read-only view 를 제공하는 것이라 목적이 다름

### Q2) Nothing을 사용했던 경험이 있으신가요? Nothing은 정말 필요한 타입일까요?
- 참고: 2장의 궁금했던 것들
  - 코틀린의 throw는 식이다. 변수의 초기화에 throw를 사용할 경우 해당 변수가 초기화되지 않는다. 하지만 '초기화되지 않는다'는 사실을 알아야 하는가?
- 나에게 필요한 것이 아님
- 컴파일러에게 필요함
  - throw 는 Nothing 타입을 반환함
    - 이 때 발생한는 '초기화되지 않는다'는 현상을 알 필요는 없음
  - 여기에 쓰임
    - Unreachable code 감지
    - smart cast에 사용
